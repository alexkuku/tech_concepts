⚙️ 配置

管理应用配置和机密信息

配置概述

Kubernetes 提供了灵活的配置管理机制，允许将配置信息与容器镜像分离，实现配置的版本控制和动态更新。

为什么需要分离配置？

  • 可重用性：同一个镜像可用于不同环境
  • 安全性：敏感信息不暴露在镜像中
  • 动态性：配置可以动态更新无需重建镜像
  • 版本控制：配置独立于代码进行版本管理

Kubernetes 配置资源

资源 适用场景
ConfigMap 存储非敏感配置数据 应用配置、环境变量、命令行参数
Secret 存储敏感配置数据 密码、证书、密钥、令牌
Downward API 注入 Pod 信息 Pod 名称、IP、标签等元数据

ConfigMap 核心概念

ConfigMap 用于存储配置数据，以键值对形式存储，可以注入到 Pod 中作为环境变量或挂载为配置文件。

ConfigMap 的特点

  • 命名空间资源：属于特定命名空间
  • 键值对存储：数据以键值对形式存储
  • 大小限制：单个 ConfigMap 最大 1MB
  • 不可加密：数据以明文存储（etcd 可以配置加密）
  • 动态更新：挂载为文件时可以动态更新

创建 ConfigMap

# 从目录创建
kubectl create configmap my-config --from-file=./config/

# 从文件创建
kubectl create configmap my-config --from-file=./app.properties

# 从键值对创建
kubectl create configmap my-config \
  --from-literal=host=example.com \
  --from-literal=port=8080

# YAML 定义
apiVersion: v1
kind: ConfigMap
metadata:
  name: game-config
data:
  player_initial_lives: "3"
  ui_properties_file_name: "user-interface.properties"
  game.properties: |
    enemy.types=aliens,monsters
    player.maximum-lives=5
  user-interface.properties: |
    color.good=purple
    color.bad=yellow
    allow.rows=20

使用 ConfigMap

# 作为环境变量
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo-pod
spec:
  containers:
    - name: demo
      image: alpine
      command: ["sleep", "3600"]
      env:
        - name: PLAYER_INITIAL_LIVES
          valueFrom:
            configMapKeyRef:
              name: game-config
              key: player_initial_lives
        - name: UI_PROPERTIES_FILE_NAME
          valueFrom:
            configMapKeyRef:
              name: game-config
              key: ui_properties_file_name

# 所有环境变量
envFrom:
  - configMapRef:
      name: game-config

# 挂载为文件
volumeMounts:
- name: config-volume
  mountPath: /etc/config
  readOnly: true
volumes:
- name: config-volume
  configMap:
    name: game-config
    items:
    - key: game.properties
      path: game.properties
    - key: user-interface.properties
      path: ui.properties

ConfigMap 更新行为

使用方式 更新行为
作为环境变量 不自动更新（需要重启 Pod）
挂载为文件 自动更新（使用 symbolic link）
最佳实践：对于需要动态更新的配置，考虑使用 Volume 方式挂载 ConfigMap。对于环境变量，需要更新后重启 Pod。

Secret 安全

Secret 用于存储敏感数据，如密码、证书、密钥等。Secret 与 ConfigMap 类似，但数据经过编码且更加安全。

Secret 类型

类型 说明 用途
Opaque 默认类型 任意用户密码/密钥
kubernetes.io/service-account-token ServiceAccount Token API 访问令牌
kubernetes.io/dockercfg Docker 配置 Docker Registry 认证
kubernetes.io/dockerconfigjson Docker 配置（JSON） Docker Registry 认证
kubernetes.io/basic-auth 基本认证 用户名/密码
kubernetes.io/ssh-auth SSH 认证 SSH 密钥
kubernetes.io/tls TLS 证书 SSL/TLS 证书
bootstrap.kubernetes.io/token Bootstrap Token 节点加入集群的令牌

创建 Secret

# 从命令行创建（自动 base64 编码）
kubectl create secret generic my-secret \
  --from-literal=password=mypassword \
  --from-literal=username=myuser

# 从文件创建
kubectl create secret tls my-tls-secret \
  --cert=path/to/cert/file \
  --key=path/to/key/file

# Docker Registry Secret
kubectl create secret docker-registry regcred \
  --docker-server=DOCKER_REGISTRY_SERVER \
  --docker-username=DOCKER_USER \
  --docker-password=DOCKER_PASSWORD

# YAML 定义
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
data:
  username: YWRtaW4=  # base64 编码的 "admin"
  password: MWYyZDFlMmU2N2Rm  # base64 编码的密码

使用 Secret

# 作为环境变量
env:
- name: SECRET_USERNAME
  valueFrom:
    secretKeyRef:
      name: my-secret
      key: username

# 挂载为文件
volumeMounts:
- name: secret-volume
  mountPath: "/etc/secrets"
  readOnly: true
volumes:
- name: secret-volume
  secret:
    secretName: my-secret
    items:
    - key: username
      path: my-group/my-username

# 拉取镜像时使用
imagePullSecrets:
- name: regcred

Secret 安全性

安全措施 说明
Base64 编码 数据经过 Base64 编码（非加密）
etcd 加密 可以配置 etcd 存储加密
RBAC 通过 RBAC 控制访问权限
最小权限 Pod 只挂载需要的 Secret
临时文件系统 Secret 挂载为 tmpfs（内存存储）
重要：Base64 不是加密！Secret 只是为了避免敏感信息出现在 Pod 定义中，不是安全存储解决方案。使用时结合 etcd 加密、RBAC 等安全措施。

探针 重要

探针（Probes）用于检测容器和 Pod 的健康状态，包括存活探针、就绪探针和启动探针。

探针类型

探针 用途 失败后果
Liveness Probe 检测容器是否正常运行 重启容器
Readiness Probe 检测容器是否准备好接收流量 从 Service 移除
Startup Probe 检测容器是否启动完成 禁用其他探针

探针配置示例

apiVersion: v1
kind: Pod
metadata:
  name: liveness-http
spec:
  containers:
  - name: liveness
    image: registry.k8s.io/liveness
    args:
    - /server
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 15  # 容器启动后等待 15 秒才开始探测
      periodSeconds: 20        # 每 20 秒探测一次
      timeoutSeconds: 1        # 超时时间 1 秒
      successThreshold: 1     # 成功 1 次即认为健康
      failureThreshold: 3     # 失败 3 次才认为不健康

探针执行方式

方式 说明 示例
ExecAction 在容器内执行命令 检查文件是否存在
HTTPGetAction 发送 HTTP GET 请求 检查 /healthz 端点
TCPSocketAction 尝试建立 TCP 连接 检查端口是否开放
gRPC (v1.24+) 发送 gRPC 健康检查 检查 gRPC 服务健康状态
探针使用建议：
  • Liveness Probe 用于检测容器是否崩溃或死锁
  • Readiness Probe 用于应用启动时间较长或依赖其他服务
  • Startup Probe 用于应用启动时间很长（保护 Liveness/Readiness 探针）

资源管理

Kubernetes 允许为 Pod 和容器指定资源请求（requests）和限制（limits），用于资源分配和调度。

资源类型

资源 说明 单位
CPU 计算资源 Cores (1000m = 1 Core)
Memory 内存资源 Bytes (Mi, Gi)
Storage (Ephemeral) 临时存储 Bytes (Mi, Gi)
Hugepages- 大页内存 固定大小页

Resource Requests 和 Limits

resources:
  requests:
    cpu: "100m"        # 请求 0.1 CPU
    memory: "128Mi"    # 请求 128MB 内存
  limits:
    cpu: "500m"        # 限制最多 0.5 CPU
    memory: "256Mi"    # 限制最多 256MB 内存

资源请求和限制的作用

设置 调度 运行时 超出行为
只有 Request 用于调度决策 使用节点可用资源 可能 OOM killed
只有 Limit Request = Limit 限制资源使用 被限制
Request = Limit 保证资源 严格限制 QoS Guarantee
Request < Limit 保证资源 有限爆发 QoS Burstable

QoS Classes

QoS 类 条件 特点
Guaranteed 所有容器的 Request = Limit 保证资源，优先级最高
Burstable 至少一个容器有 Request，但不是 Guaranteed 可以使用额外资源，优先级中等
BestEffort 没有设置 Request 和 Limit 尽力而为，优先级最低

Downward API

Downward API 允许容器获取自身和 Pod 的元数据，如 Pod 名称、IP、标签等。

Downward API 功能

功能 说明
FieldRef 引用 Pod 的字段
ResourceFieldRef 引用容器资源请求/限制
ConfigMapKeyRef 引用 ConfigMap 中的值
SecretKeyRef 引用 Secret 中的值

Downward API 示例

# 作为环境变量
env:
- name: MY_POD_NAME
  valueFrom:
    fieldRef:
      fieldPath: metadata.name
- name: MY_POD_IP
  valueFrom:
    fieldRef:
      fieldPath: status.podIP
- name: MY_CPU_REQUEST
  valueFrom:
    resourceFieldRef:
      resource: requests.cpu

# 挂载为文件
volumeMounts:
- name: podinfo
  mountPath: /etc/podinfo
  readOnly: true
volumes:
- name: podinfo
  downwardAPI:
    items:
    - path: "labels"
      fieldRef:
        fieldPath: metadata.labels
    - path: "annotations"
      fieldRef:
        fieldPath: metadata.annotations

本章总结

关键要点：
  • ConfigMap 用于存储非敏感配置，可以注入为环境变量或文件
  • Secret 用于存储敏感信息，数据存储时要结合安全措施
  • 探针用于健康检查，包括 Liveness、Readiness 和 Startup 探针
  • 资源 Request 和 Limits 用于资源分配和调度
  • QoS 类包括 Guaranteed、Burstable 和 BestEffort
  • Downward API 允许容器获取自身和 Pod 的元数据
  • 配置管理实现了配置与镜像的分离，提高了应用的可维护性
