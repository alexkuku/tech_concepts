Kubernetes 集群架构概述

Kubernetes 集群由一组运行容器化应用的机器（节点）组成。集群分为两个主要部分：控制平面（Control Plane）和工作节点（Worker Nodes）。

控制平面 (Control Plane)
↑ 管理/协调 ↑
↓ 通信/监控 ↓
工作节点 1
工作节点 2
工作节点 3
集群核心特点：
  • 分布式架构：控制平面和工作节点可以分布在多台机器上
  • 高可用性：控制平面组件可以部署多个副本以提高可用性
  • 可扩展性：可以动态添加或移除工作节点
  • 自愈能力：组件故障时自动恢复

控制平面详解

控制平面（也称 Master 节点）是 Kubernetes 集群的"大脑"，负责集群的决策、管理和协调。它包含以下核心组件：

etcd (存储)
↕
API Server (入口)
↓
Scheduler
Controller Manager
Cloud Controller

1. API Server (kube-apiserver)

核心作用：API Server 是整个 Kubernetes 系统的统一入口，是所有组件（内部和外部）与集群通信的唯一接口。
功能 说明
RESTful API 提供 REST 风格的 API 接口，支持 JSON 和 Protobuf 格式
认证和授权 对所有请求进行身份验证和权限检查
准入控制 通过准入控制器对请求进行扩展和验证
数据验证 验证请求的数据格式和有效性
持久化 将状态数据持久化到 etcd

2. etcd

核心作用：etcd 是 Kubernetes 的后端数据库，存储整个集群的配置和状态信息，是集群的"单一事实来源"。
特性 说明
键值存储 使用键值对方式存储数据
一致性 基于 Raft 协议，保证数据强一致性
高可用 支持集群模式，可以部署多个副本
Watch 机制 支持监听数据变化，通知订阅者
事务支持 支持多键事务操作

etcd 存储的数据类型

  • 节点信息（Node 对象）
  • Pod 信息（Pod 对象）
  • 服务信息（Service 对象）
  • 配置信息（ConfigMap、Secret 对象）
  • 存储信息（PV、PVC 对象）
  • 集群配置和策略
  • 用户和权限信息

3. Scheduler (kube-scheduler)

核心作用：Scheduler 负责调度 Pod，将新创建的 Pod 分配到合适的节点上运行。

调度流程

1. 监听未绑定的 Pod
↓
2. 过滤不合适的节点
↓
3. 对候选节点打分
↓
4. 选择最高分节点
↓
5. 绑定 Pod 到节点
调度策略 说明
资源请求 根据 Pod 的 CPU、内存需求选择有足够资源的节点
节点选择器 根据 nodeSelector 匹配节点的标签
污点和容忍度 根据节点上的污点和 Pod 的容忍度进行匹配
亲和性/反亲和性 根据 Pod 的亲和性和反亲和性规则调度
资源利用率 优先选择资源利用率较均衡的节点
拓扑传播约束 控制 Pod 在不同拓扑域（如机架、可用区）的分布

4. Controller Manager (kube-controller-manager)

核心作用：Controller Manager 运行控制器进程，监控集群状态并通过 API Server 进行调整，确保集群实际状态与期望状态一致。

包含的控制器

控制器 功能
Node Controller 监控节点状态，处理节点故障和心跳
Replication Controller 确保指定数量的 Pod 副本在运行
Endpoints Controller 维护 Service 的后端端点列表
Service Account Controller 为命名空间创建默认的 ServiceAccount
Token Controller 创建 API 访问令牌
CronJob Controller 管理定时任务
Deployment Controller 管理 Deployment 和执行滚动更新
StatefulSet Controller 管理有状态应用
DaemonSet Controller 确保每个节点运行所需 Pod
Job Controller 管理一次性任务

↓

5. Cloud Controller Manager

核心作用：Cloud Controller Manager 是云服务提供商的接口，将 Kubernetes 与底层云平台集成。
控制器 支持的云平台 功能
Node Controller 所有平台 管理云主机节点的生命周期
Route Controller AWS, GCE 配置云平台的路由表
Service Controller AWS, GCE, Azure 创建云平台的负载均衡器
Volume Controller AWS, GCE, Azure 创建和挂载云平台的存储卷
注意：如果你的集群运行在裸金属服务器上（不使用任何云平台），可以不部署 Cloud Controller Manager。

工作节点详解

工作节点（Worker Node）是 Kubernetes 集群中运行容器化应用的工作机。每个工作节点都运行以下组件：

工作节点
↓ 包含 ↓
Kubelet
Kube-proxy
Container Runtime
↓ 运行 ↓
Pods (容器)

1. Kubelet

核心作用：Kubelet 是节点的主要代理，负责与控制平面通信，管理节点上的 Pod 和容器。
功能 说明
接收指令 从控制平面接收 Pod 分配和更新指令
管理容器 通过容器运行时创建、启动、停止和删除容器
健康检查 定期向控制平面汇报节点和 Pod 的健康状态
监控资源 监控节点的 CPU、内存、磁盘等资源使用情况
挂载卷 为 Pod 挂载所需的存储卷
垃圾回收 清理未使用的容器、镜像等资源

Kubelet 工作流程

1. 向 API Server 注册节点
↓
2. 定期同步节点状态
↓
3. 接收 Pod 分配
↓
4. 通过 CRI 创建容器
↓
5. 监控容器状态
↓
6. 上报 Pod 状态

2. Kube-proxy

核心作用：Kube-proxy 是节点的网络代理，负责维护节点上的网络规则，实现 Service 的负载均衡。

Kube-proxy 工作模式

模式 实现方式 优点 缺点
Userspace 用户空间代理 兼容性好 性能差，已弃用
iptables 使用 iptables 规则 成熟稳定，广泛使用 大量规则时性能下降
ipvs 使用 IPVS 负载均衡 性能高，支持复杂调度 配置稍复杂
Kernelspace 内核空间代理 性能最高 需要内核支持

Kube-proxy 主要功能

  • 服务发现：监听 Service 和 Endpoints 的变化
  • 负载均衡：将流量分发到后端 Pod
  • 网络规则：维护 iptables 或 ipvs 规则
  • 健康检查：移除不健康的后端

3. Container Runtime (容器运行时)

核心作用：容器运行时是真正运行容器的软件，负责下载镜像、创建容器、管理容器生命周期。

支持的容器运行时

运行时 状态 特点
Docker ⚠️ 社区版已弃用 最常见，通过 dockershim 兼容
containerd ✓ 推荐 轻量、高效、稳定
CRI-O ✓ 推荐 专为 Kubernetes 设计，轻量
gVisor ↑ 发展中 提供安全沙箱环境
Kata Containers ↑ 发展中 基于虚拟机的容器运行时

CRI (Container Runtime Interface)

CRI 是 Kubernetes 与容器运行时之间的标准接口。通过 CRI，Kubernetes 可以与任何实现了该接口的容器运行时配合工作。

节点与控制平面之间的通信

Kubernetes 集群中的各组件需要相互通信才能正常工作。通信主要通过以下方式：

控制平面
↑ HTTPS REST API ↑
↓ 心跳/状态上报 ↓
工作节点 (Kubelet)

通信模式

方向 发起方 接收方 协议 内容
控制平面 → 节点 API Server Kubelet HTTPS Pod 分配、更新、删除指令
节点 → 控制平面 Kubelet API Server HTTPS 节点状态、Pod 状态、心跳
调度器 → 节点 Scheduler Kubelet 通过 API Server Pod 绑定信息
控制平面 → 节点 Controller Manager Kube-proxy 通过 API Server Service 和 Endpoints 变化
通信安全：
  • 所有控制平面到节点的通信都使用 HTTPS 加密
  • 使用 TLS 证书验证身份
  • 支持双向认证（MTLS）
  • 可以通过 ServiceAccount 进行细粒度权限控制

租约 (Leases)

租约是 Kubernetes 中的一种机制，用于表示分布式系统中领导者选举和心跳的协调原语。

租约的主要用途

  • 节点心跳：kubelet 使用租约向控制平面汇报节点存活
  • 控制器领导选举：多个控制器副本使用租约选举主控制器
  • 锁机制：防止多个组件同时执行某个操作

租约 vs 心跳机制

特性 传统心跳 租约 (Leases)
方式 更新 Node 对象状态 专用的 Lease 对象
性能 较慢（需要更新整个对象） 更快（只更新 lease 时间）
资源开销 较大 小
推荐程度 已弃用 推荐使用

Kubernetes 自愈能力

Kubernetes 具有强大的自愈能力，能够在节点或 Pod 故障时自动恢复。自愈能力通过多个层次的控制器和监控机制实现。

故障发生
↓
检测到失败（心跳超时/健康检查失败）
↓
控制器识别状态差异
↓
执行恢复操作
↓
系统恢复健康状态

自愈场景

故障类型 检测机制 恢复方式
节点故障 心跳超时，Node Controller 将标记为 NotReady，重新调度 Pod
容器崩溃 Kubelet 健康检查 根据重启策略重启容器
Pod 被删除 ReplicaSet/Deployment 创建新的 Pod 副本
应用未通过健康检查 Liveness Probe 重启容器
应用未就绪 Readiness Probe 从 Service 中移除，等待就绪
控制平面组件故障 健康检查 自动重启（如果正确配置）
自愈的关键设计：
  • 声明式 API：只需要声明期望状态，系统自动维护实际状态
  • 持续监控：控制器持续 Watch 资源变化
  • 自动重试：失败的操作会自动重试
  • 故障隔离：故障不会传播到整个集群

垃圾回收

垃圾回收是指 Kubernetes 自动清理不再使用的资源，以防止资源泄漏和浪费。

垃圾回收类型

类型 回收内容 触发条件
容器垃圾回收 已停止的容器 超过存活时间或达到数量限制
镜像垃圾回收 未被使用的镜像 磁盘空间不足或达到阈值
对象垃圾回收 被删除对象的依赖 所有者对象被删除
已完成的 Job 已完成的 Pod Job 完成且超过保留时间

删除策略

类型 说明 示例
Foreground 先删除依赖对象，再删除所有者 Deleting Pod → 删除 Pod → 删除 Deployment
Background 立即删除所有者，后台删除依赖 删除 Deployment → 立即删除 → 后台删除所有 Pod
Orphan 只删除所有者，保留依赖对象 删除 Deployment → 只删除 Deployment → 保留 Pod

所有者和依赖者

在 Kubernetes 中，对象可以建立所有者-依赖者关系（也称为父子关系）。

# Pod 引用 Deployment 作为所有者
metadata:
  ownerReferences:
  - apiVersion: apps/v1
    kind: Deployment
    name: nginx-deployment
    uid: 7d9c1a2b-3e4f-5g6h-7i8j-9k0l1m2n3o4p
    controller: true
    blockOwnerDeletion: true
字段说明：
  • controller: true：表示此对象是控制器的从属对象
  • blockOwnerDeletion: true：禁止删除所有者对象（当依赖对象还存在时）

本章总结

关键要点：
  • Kubernetes 集群由控制平面和工作节点组成，通过 API Server 协调工作
  • 控制平面包含 API Server、etcd、Scheduler、Controller Manager 和 Cloud Controller Manager
  • 工作节点包含 Kubelet、Kube-proxy 和容器运行时
  • API Server 是整个系统的统一入口，所有通信都通过它进行
  • etcd 存储集群的所有状态数据，是单一事实来源
  • Scheduler 负责 Pod 调度，Controller Manager 通过控制器确保期望状态与实际状态一致
  • Kubelet 管理节点上的容器，Kube-proxy 实现服务负载均衡
  • Kubernetes 具有强大的自愈能力，会自动恢复故障
  • 垃圾回收机制确保资源不会泄漏
