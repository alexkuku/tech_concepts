工作负载概述

工作负载（Workloads）是 Kubernetes 中运行应用程序的对象。Kubernetes 提供了多种工作负载资源来满足不同类型应用的需求。

工作负载的核心价值：
  • 管理 Pod 的生命周期和副本数量
  • 支持应用的部署、更新和回滚
  • 提供自愈能力和高可用性
  • 支持批处理任务和定时任务

工作负载类型对比

工作负载 适用场景 状态 网络标识
Deployment 无状态应用（Web、API） 无状态 动态 DNS
StatefulSet 有状态应用（数据库、MQ） 有状态 固定 DNS
DaemonSet 系统守护进程 无状态/有状态 无
Job 一次性任务、批处理 无状态 无
CronJob 定时任务、备份 无状态 无

Pod 核心概念

Pod 是 Kubernetes 中最小和最简单的部署单元。一个 Pod 可以包含一个或多个容器，这些容器共享网络和存储命名空间。

Pod 的特性

  • 共享网络：同一 Pod 中的容器共享同一个网络命名空间（相同的 IP 和端口空间）
  • 共享存储：容器可以通过 Volume 共享数据
  • 原子性：Pod 通常作为一个整体被调度和终止
  • 临时性：Pod 是临时的，可以被任意创建、销毁和替换
  • 唯一 IP：每个 Pod 都有一个独立的集群 IP 地址

Pod 状态详解

相位 说明 常见原因
Pending Pod 已被接受但容器未创建 等待调度、镜像拉取中
Running 至少一个容器在运行 正常状态
Succeeded 所有容器成功终止 Job 完成任务
Failed 至少一个容器异常终止 应用错误、OOM
Unknown 无法确定 Pod 状态 节点通信失败

Pod 模式

模式 说明 示例
单容器 Pod Pod 中只运行一个主容器 Nginx Web 服务器
多容器 Pod (Sidecar) 主容器 + 辅助容器（日志、监控） Web 应用 + 日志收集器
Init 容器 在主容器前运行的初始化容器 初始化数据库、下载配置
临时容器 调试时临时添加到 Pod 的容器 调试工具、诊断程序

Init 容器

Init 容器（Init Containers）是一种专用容器，在 Pod 主容器启动之前运行。它们可以包含配置或初始化应用的实用程序。

Init 容器的特性

  • 顺序执行：多个 Init 容器按顺序依次执行
  • 成功即退出：Init 容器必须成功退出，主容器才会启动
  • 独立运行：与主容器有不同的文件系统和配置
  • 失败重试：失败后会重试，直到成功

Pod 中断

Pod 中断是指 Pod 被主动删除、驱逐或排空的情况。Kubernetes 提供 Pod Disruption Budget（PDB）来保护关键应用。

中断类型

类型 说明 示例
自愿中断 主动触发的删除 删除 Deployment、节点维护
非自愿中断 意外或强制中断 资源不足、硬件故障

Pod 中断预算 (PDB)

PDB 限制了同时中断的 Pod 数量，确保应用的高可用性。

# 定义 Pod 中断预算
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: nginx-pdb
spec:
  minAvailable: 2  # 至少保持 2 个可用 Pod
  # maxUnavailable: 1  # 或者最多允许 1 个不可用
  selector:
    matchLabels:
      app: nginx

PDB 参数

参数 说明 示例
minAvailable 最少可用 Pod 数量 minAvailable: 2 - 至少保持 2 个可用
maxUnavailable 最多不可用 Pod 数量 maxUnavailable: 1 - 最多 1 个不可用

Deployment 最常用

Deployment 用于管理无状态应用，管理 ReplicaSet，支持应用的部署、更新和回滚。

Deployment 的特性

  • 声明式更新：声明期望状态，自动实现变更
  • 滚动更新：渐进式更新，零停机部署
  • 回滚能力：支持回滚到任何历史版本
  • 扩展能力：支持自动扩展 Pod 数量
  • 自动恢复：Pod 故障时自动重建

ReplicaSet

Pod 2

滚动更新策略

spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # 最多可以比期望多 1 个 Pod
      maxUnavailable: 0     # 更新时最多允许 0 个不可用（推荐）
  minReadySeconds: 10      # Pod 就绪后等待 10 秒才视为可用

更新策略对比

策略 说明 特点
RollingUpdate 滚动更新，渐进式替换 零停机，默认策略
Recreate 先删除所有 Pod，再创建新 Pod 停机时间短，一次性替换

常用命令

操作 命令
创建 Deployment kubectl create deployment nginx --image=nginx
查看 Deployment 状态 kubectl rollout status deployment/nginx
查看更新历史 kubectl rollout history deployment/nginx
回滚到上一版本 kubectl rollout undo deployment/nginx
回滚到指定版本 kubectl rollout undo deployment/nginx --to-revision=2
扩缩容 kubectl scale deployment nginx --replicas=5

StatefulSet

StatefulSet 用于管理有状态应用，为 Pod 提供稳定的唯一标识、稳定的持久化存储和有序的部署和扩展。

StatefulSet 的特性

  • 稳定网络标识：每个 Pod 有唯一的、稳定的 DNS 名称
  • 有序部署：Pod 按顺序启动（0 → 1 → 2...）
  • 有序扩展：扩容时按顺序添加 Pod
  • 有序收缩：缩容时按逆序删除 Pod
  • 持久化存储：每个 Pod 关联独立的 PersistentVolume
StatefulSet: mysql
Pod: mysql-0
Pod: mysql-1
Pod: mysql-2
PV: mysql-0
PV: mysql-1
PV: mysql-2

StatefulSet vs Deployment

特性 Deployment StatefulSet
Pod 名称 随机哈希值 有序索引（0, 1, 2...）
DNS 名称 共享 Service DNS 独立的 Headless Service DNS
启动顺序 无序，并行启动 有序，依次启动
存储 共享或无 每个 Pod 独立的 PV
更新策略 滚动更新 滚动更新（自动分区）
适用场景 无状态应用 有状态应用（数据库、MQ）

使用场景

  • 数据库服务（MySQL、PostgreSQL、MongoDB）
  • 消息队列（Kafka、RabbitMQ）
  • 分布式集群（Redis Cluster、Elasticsearch）
  • 任何需要稳定网络标识和持久化存储的应用

DaemonSet

DaemonSet 确保在所有（或部分）节点上运行一个 Pod 副本，常用于系统级守护进程。

DaemonSet 的特性

  • 每节点一个：每个节点运行一个 Pod
  • 自动调度：新节点加入时自动创建 Pod
  • 节点选择：可以通过节点选择器选择特定节点
  • 容忍污点：可以自动容忍节点的污点

Job 和 CronJob

Job 用于管理一次性任务，CronJob 用于定时执行任务。

Job 的类型

类型 配置 说明
非并行 Job completions: 1 只运行一个 Pod，完成即结束
固定完成次数 Job completions: N 运行 N 个 Pod，完成 N 个作业
并行 Job parallelism: N 并行运行 N 个 Pod，每个处理一部分

Backoff 策略

策略 说明
OnFailure 失败时重启容器
Never 失败时不重启，重新创建 Pod

CronJob 配置

apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "*/1 * * * *"  # 每分钟执行一次
  successfulJobsHistoryLimit: 3  # 保留 3 个成功的 Job
  failedJobsHistoryLimit: 1      # 保留 1 个失败的 Job
  jobTemplate:
    spec:
      completions: 1        # 需要 1 次成功
      parallelism: 1        # 并行度为 1
      backoffLimit: 3       # 最多重试 3 次
      activeDeadlineSeconds: 600  # Job 超时时间
      template:
        spec:
          containers:
          - name: hello
            image: busybox
            imagePullPolicy: IfNotPresent
            command:
            - /bin/sh
            - -c
            - date; echo Hello from the Kubernetes cluster
          restartPolicy: OnFailure

Cron 表达式格式

字段 有效值 说明
分钟 0-59 每小时的第几分钟
小时 0-23 每天的第几小时
日 1-31 每月的第几天
月 1-12 第几月
星期 0-6 星期几（0 或 7 为周日）

常用 Cron 表达式

表达式 说明
*/5 * * * * 每 5 分钟执行一次
0 */2 * * * 每 2 小时执行一次
0 0 * * * 每天凌晨执行一次
0 0 * * 0 每周日执行一次
0 0 1 * * 每月 1 号执行一次

自动扩缩容

Kubernetes 提供了水平 Pod 自动扩缩容（HPA）和垂直 Pod 自动扩缩容（VPA）来动态调整应用规模。

HPA (Horizontal Pod Autoscaler)

HPA 根据资源使用率自动调整 Deployment 的副本数量。

# HPA 示例
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 80  # CPU 使用率达到 80% 时扩容
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80

HPA 指标类型

指标类型 说明 示例
Resource 基于 CPU/内存使用率 CPU 使用率 > 80%
Pods 基于 Pod 的自定义指标 每秒请求数
Object 基于集群或外部指标 Ingress QPS

VPA (Vertical Pod Autoscaler)

VPA 自动调整 Pod 的 CPU 和内存请求及限制。

VPA 特性：
  • 自动调整资源请求（request）和限制（limit）
  • 支持关闭和重启 Pod（Off 模式、Initial 模式、Recreate 模式、Auto 模式）
  • 与 HPA 不兼容（不能同时用于 CPU/内存）

本章总结

关键要点：
  • Pod 是 Kubernetes 中最小的部署单元，可以包含一个或多个容器
  • Init 容器在主容器前运行，用于初始化和准备环境
  • PDB 保护关键应用，限制同时中断的 Pod 数量
  • Deployment 是最常用的控制器，支持滚动更新和回滚
  • StatefulSet 用于有状态应用，提供稳定的网络标识和持久化存储
  • DaemonSet 确保每个节点运行一个 Pod，适用于系统守护进程
  • Job 管理一次性任务，CronJob 用于定时执行任务
  • HPA 自动调整 Pod 副本数，VPA 自动调整 Pod 资源
