🐳 容器

理解 Kubernetes 中的容器运行和生命周期

容器概述

容器是一种轻量级、独立、可执行的软件包，包含了运行应用程序所需的所有内容：代码、运行时、系统工具、系统库和设置。

Kubernetes 中的容器：
  • 容器是 Kubernetes 中最小的可部署单元
  • 容器通常运行在 Pod 中，一个 Pod 可以包含一个或多个容器
  • Kubernetes 通过容器运行时接口（CRI）与容器运行时交互
  • 支持多种容器运行时实现（containerd、CRI-O、Docker 等）

容器 vs 虚拟机

特性 容器 虚拟机
抽象级别 应用级 操作系统级
资源开销 小（共享主机内核） 大（独立操作系统）
启动速度 快（秒级） 慢（分钟级）
隔离性 进程级 硬件级
便携性 高（镜像标准化） 中（需要匹配虚拟化平台）
密度 高（可运行数百个容器） 低（通常运行几个虚拟机）

容器镜像

容器镜像是容器的只读模板，包含了运行容器所需的所有文件和配置。当你运行一个容器时，Kubernetes 会使用指定的容器镜像来创建容器实例。

镜像格式

  • Docker 镜像：最常用的镜像格式，使用 OCI 规范
  • OCI 镜像：开放容器倡议的标准镜像格式
  • Appc 镜像：较早的格式，支持有限

镜像拉取策略

策略 值 说明
Always imagePullPolicy: Always 总是拉取最新的镜像
IfNotPresent imagePullPolicy: IfNotPresent 本地存在则使用，否则拉取（默认）
Never imagePullPolicy: Never 只使用本地镜像，从不拉取

私有镜像仓库认证

# 创建 Docker Registry Secret
kubectl create secret docker-registry regcred \
  --docker-server=<your-registry-server> \
  --docker-username=<your-name> \
  --docker-password=<your-pword> \
  --docker-email=<your-email>

# 在 Pod 中引用
spec:
  containers:
  - name: app
    image: myregistry.example.com/myapp:1.0
  imagePullSecrets:
  - name: regcred

镜像命名规范

格式 示例 说明
tag:latest nginx:latest 最新版本标签
特定版本 nginx:1.21.6 指定具体版本
摘要 nginx@sha256:abc123... 使用 Content Addressable Storage (CAS)
私有仓库 registry.example.com/nginx:1.0 指定仓库地址
命名空间 my-org/nginx:1.0 Docker Hub 命名空间
最佳实践：生产环境中应避免使用 :latest标签，因为它会导致不可预测的行为。应使用具体的版本标签或镜像摘要。

容器环境

Kubernetes 为容器提供了一个丰富的运行环境，包括环境变量、挂载卷、资源限制等配置。

注入的环境变量

环境变量 示例 说明
KUBERNETES_SERVICE_HOST 10.96.0.1 Kubernetes API Server 的主机地址
KUBERNETES_SERVICE_PORT 443 Kubernetes API Server 的端口号
POD_NAME nginx-deployment-abc123 当前 Pod 的名称
POD_NAMESPACE default 当前 Pod 的命名空间
POD_IP 10.244.1.5 当前 Pod 的 IP 地址
NODE_NAME node-1 运行当前 Pod 的节点名称

环境变量配置方式

# 直接设置值
env:
  - name: DATABASE_URL
    value: "postgres://db:5432/mydb"

# 从 ConfigMap 引入
envFrom:
  - configMapRef:
      name: app-config

# 从 Secret 引入
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-secret
        key: password

# 使用 Downward API
env:
  - name: MY_POD_IP
    valueFrom:
      fieldRef:
        fieldPath: status.podIP

容器文件系统

  • 只读根文件系统：可以提高安全性
  • 挂载卷：通过 volumeMounts 挂载到容器内部
  • 临时文件系统：使用 emptyDir 类型卷
  • 共享存储：多个容器可以共享同一个卷

运行时类 (RuntimeClass)

RuntimeClass 是 Kubernetes 中用于定义不同容器运行时配置的资源。它允许你在集群中使用多种容器运行时，而无需修改 Pod 定义。

RuntimeClass 的用途

  • 多运行时支持：同时使用 containerd、gVisor、Kata Containers 等
  • 配置隔离：为不同工作负载提供不同的运行时配置
  • 安全要求：为敏感应用使用强隔离的运行时
  • 性能优化：为高性能应用使用轻量级运行时

RuntimeClass 定义示例

# RuntimeClass 定义
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc
overhead:
  podFixed:
    memory: "100Mi"
    cpu: "100m"

# Pod 使用 RuntimeClass
apiVersion: v1
kind: Pod
metadata:
  name: mypod
spec:
  runtimeClassName: gvisor
  containers:
  - name: mycontainer
    image: nginx

常见运行时

运行时 Handler 名称 特点 适用场景
runc runc 标准运行时，Docker 默认 通用容器应用
gVisor runsc 用户空间内核，提供沙箱 不可信应用、多租户
Kata Containers kata-qemu 基于虚拟机的隔离 高安全要求、PCI-DSS 合规
crun crun 轻量级、C 实现 资源受限环境

容器生命周期

了解容器的生命周期对于管理应用非常重要。Kubernetes 管理容器的整个生命周期，从创建到删除。

容器状态

状态 说明 常见原因
Waiting 容器正在启动或等待条件 镜像拉取中、启动命令执行中
Running 容器正在运行 容器启动成功，主进程在运行
Terminated 容器已停止 正常退出、被杀掉、OOM

容器退出原因

原因 说明 退出码
Completed 容器正常退出 0
Error 容器因错误退出 非 0
OOMKilled 被 OOM Killer 杀掉 137
ContainerCannotRun 容器运行时错误 125-127
StartError 容器启动错误 128

重启策略

策略 说明 适用场景
Always 总是重启容器 长期运行的服务
OnFailure 仅在失败时重启 可能正常退出的任务
Never 从不重启 批处理作业、Job
注意：重启策略应用于 Pod 内的所有容器。如果一个 Pod 包含多个容器，所有容器的重启策略必须一致。

容器生命周期钩子 重要

容器生命周期钩子允许你在容器生命周期的特定时刻执行操作。Kubernetes 提供了两种钩子：

钩子类型

钩子 触发时机 用途
PostStart 容器启动后立即触发 预热、初始化配置、注册服务
PreStop 容器终止前触发 优雅停机、清理资源、注销服务

钩子执行方式

# 使用 exec 执行命令
lifecycle:
  postStart:
    exec:
      command:
      - /bin/sh
      - -c
      - echo "Container started" > /tmp/start.log

# 使用 HTTP 请求
lifecycle:
  preStop:
    httpGet:
      path: /shutdown
      port: 8080
      host: localhost

# 使用 TCP 套接字
lifecycle:
  preStop:
    tcpSocket:
      port: 8080
      host: localhost

钩子特性

重要特性：
  • 同步执行：钩子同步执行且必须完成才能继续容器生命周期
  • 超时限制：默认超时时间 1 秒，可通过 terminationGracePeriodSeconds 配置
  • 失败处理：钩子失败导致容器终止（PreStop 会阻止容器被删除）
  • 日志记录：钩子输出会记录在容器日志中
  • 幂等性：钩子可能被多次调用，应确保操作是幂等的

探针 核心概念

探针是 Kubernetes 用于检测容器状态的机制。通过探针，Kubernetes 可以知道容器是否健康、是否准备好接收流量。

探针类型

探针 用途 失败后果
Liveness Probe 检测容器是否活着 重启容器
Readiness Probe 检测容器是否准备好接收流量 从 Service 移除
Startup Probe 检测容器是否已经启动 禁用存活和就绪探针

探针执行方式

方式 说明 示例
ExecAction 在容器内执行命令 exec: command: [cat, /tmp/healthy]
HTTPGetAction 发送 HTTP GET 请求 httpGet: path: /healthz, port: 8080
TCPSocketAction 尝试建立 TCP 连接 tcpSocket: port: 8080
gRPC 发送 gRPC 请求（K8s 1.24+） grpc: port: 9000, service: healthcheck

探针配置示例

# HTTP 探针示例
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    httpHeaders:
    - name: Custom-Header
      value: Awesome
  initialDelaySeconds: 10
  periodSeconds: 5
  timeoutSeconds: 1
  successThreshold: 1
  failureThreshold: 3

# 命令探针示例
readinessProbe:
  exec:
    command:
    - cat
    - /tmp/ready
  initialDelaySeconds: 5
  periodSeconds: 10

# 启动探针示例（慢启动应用）
startupProbe:
  httpGet:
    path: /startup
    port: 8080
  failureThreshold: 30  # 最多允许失败 30 次
  periodSeconds: 5       # 每 5 秒检查一次
  # 总共允许 150 秒启动

探针参数说明

参数 默认值 说明
initialDelaySeconds 0 容器启动后等待多久开始第一次探测
periodSeconds 10 执行探测的间隔时间
timeoutSeconds 1 探测的超时时间
successThreshold 1 探测成功的最小连续次数
failureThreshold 3 探测失败的最小连续次数才视为失败
terminationGracePeriodSeconds 30 容器终止时的宽限期
探针使用场景：
  • Liveness Probe：检测应用是否崩溃或死锁，重启容器恢复服务
  • Readiness Probe：应用启动时间较长或依赖其他服务时使用
  • Startup Probe：应用启动时间很长（加载大量数据）时保护存活和就绪探针

容器运行时接口 (CRI)

容器运行时接口（Container Runtime Interface，CRI）是 Kubernetes 中的一组插件接口，用于集成第三方容器运行时。

CRI 架构

Kubelet
↑ 通过 gRPC CRI API ↑
CRI Shim (dockershim 已弃用)
↑ 调用 ↑
containerd
CRI-O
其他 CRI 实现

CRI 主要功能

接口组 功能
RuntimeService Pod/容器生命周期管理、状态查询、Exec/Attach/PortForward
ImageService 镜像拉取、列出、删除、状态查询

CRI 历史变迁

阶段 变化 影响
v1.24 之前 使用 dockershim（内置 Docker 集成） Docker 是最常用运行时
v1.24 移除 dockershim 需要使用 CRI-compliant 运行时或 cri-dockerd
v1.25+ 完全基于 CRI 规范 containerd、CRI-O 成为主流

本章总结

关键要点：
  • 容器是 Kubernetes 中最小部署单元，轻量、高效、可移植
  • 容器镜像是只读模板，支持多种拉取策略和私有仓库认证
  • Kubernetes 为容器提供丰富的环境变量和配置选项
  • RuntimeClass 允许集群使用多种容器运行时
  • 容器生命周期包括 Waiting、Running、Terminated 三种状态
  • 生命周期钩子（PostStart/PreStop）允许在特定时刻执行操作
  • 探针（Liveness/Readiness/Startup）是健康检查的核心机制
  • CRI 是 Kubernetes 与容器运行时之间的标准接口
