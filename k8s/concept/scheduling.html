📊 调度

理解 Pod 调度和资源分配

调度概述

Kubernetes 调度器负责将 Pod 分配到合适的节点上运行。调度决策基于资源需求、策略约束、亲和性/反亲和性规则等多种因素。

调度的核心目标

  • 资源分配：确保 Pod 有足够的资源运行
  • 约束满足：满足各种调度约束和策略
  • 负载均衡：均匀分布工作负载
  • 高可用：分散应用提高可用性
  • 效率：最大化资源利用率

调度流程

Kubernetes 调度器通过两个阶段完成调度：过滤（Filtering）和打分（Scoring）。

调度流程详解

1. Watch 待调度 Pod
↓
2. 预选阶段：过滤不合适的节点
↓
3. 优选阶段：对候选节点打分
↓
4. 选择最高分节点
↓
5. 绑定 Pod 到节点

预选阶段

预选阶段根据预设的规则过滤掉不满足条件的节点。

调度器扩展 说明
NodeUnschedulable 检查节点是否标记为不可调度
NodeResourcesFit 检查节点是否有足够资源
NodePorts 检查节点是否有所需端口
NodeAffinity 检查节点亲和性约束
TaintToleration 检查污点和容忍度
NodeVolumeLimits 检查卷限制
VolumeZone 检查卷可用区匹配
PodToleratesNodeTaints 检查 Pod 是否容忍节点污点

优选阶段

优选阶段对通过预选的节点进行评分，选择最高分的节点。

评分策略 说明
NodeResourcesFit 资源利用率越均衡分数越高
NodeAffinity 根据亲和性规则加分
TaintToleration 根据容忍度加分
ImageLocality 镜像已存在的节点分数更高
InterPodAffinity 根据 Pod 间亲和性加分
NodeAntiAffinity 根据反亲和性加分或减分

节点选择器

节点选择器（NodeSelector）是最简单的节点选择方式，通过标签选择器匹配节点。

使用方式

# 为节点打标签
kubectl label nodes node1 disktype=ssd

# Pod 使用节点选择器
apiVersion: v1
kind: Pod
metadata:
  name: pod-with-nodeselector
spec:
  nodeSelector:
    disktype: ssd  # 只有有 disktype=ssd 标签的节点才会被调度
  containers:
  - name: nginx
    image: nginx

限制

  • 只能基于标签的精确匹配
  • 不支持复杂的条件表达式
  • 不支持优先级或软约束

节点亲和性 重要

节点亲和性（Node Affinity）提供了更强大的节点选择机制，支持硬约束（Must）和软约束（Prefer）。

亲和性类型

类型 说明 必须满足
requiredDuringSchedulingIgnoredDuringExecution 硬约束，调度时必须满足 是
preferredDuringSchedulingIgnoredDuringExecution 软约束，尽量满足 否

亲和性配置示例

apiVersion: v1
kind: Pod
metadata:
  name: with-node-affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values:
            - node1
            - node2
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        preference:
          matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx

操作符

操作符 说明 示例
In 标签值在列表中 disktype in (ssd, hdd)
NotIn 标签值不在列表中 disktype notin (ssd)
Exists 标签存在 disktype
DoesNotExist 标签不存在 !disktype
Gt 标签值大于指定值 memory > 16Gi
Lt 标签值小于指定值 memory < 32Gi

Pod 亲和性和反亲和性

Pod 亲和性根据已运行的 Pod 的标签来调度新 Pod。

Pod 亲和性类型

类型 说明
Pod Affinity 将 Pod 调度到有/没有特定 Pod 的节点
Pod Anti-Affinity 避免将 Pod 调度到有特定 Pod 的节点

Pod 亲和性示例

# Pod 亲和性：与带有 app=frontend 标签的 Pod 在同一个节点
apiVersion: v1
kind: Pod
metadata:
  name: with-pod-affinity
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - frontend
        topologyKey: kubernetes.io/hostname  # 拓扑域
  containers:
  - name: nginx
    image: nginx
# Pod 反亲和性：不在同一个节点
podAntiAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
  - labelSelector:
      matchExpressions:
      - key: app
        operator: In
        values:
        - redis
    topologyKey: "kubernetes.io/hostname"
拓扑域（Topology Key）定义了调度的范围。常用拓扑域包括：
  • kubernetes.io/hostname：节点级
  • topology.kubernetes.io/zone：可用区级
  • topology.kubernetes.io/region：区域级

污点和容忍度 重要

污点（Taint）和容忍度（Toleration）是节点级别的调度约束，用于控制哪些 Pod 可以调度到节点。

污点和容忍度的概念

污点是标记在节点上的属性，排斥不匹配容忍度的 Pod。

容忍度是 Pod 的属性，允许 Pod 调度到有特定污点的节点。

污点示例

# 为节点添加污点
kubectl taint nodes node1 key=value:NoSchedule
kubectl taint nodes node1 key=value:PreferNoSchedule
kubectl taint nodes node1 key=value:NoExecute

# 查看节点污点
kubectl describe node node1 | grep Taint

污点效果（Taint Effect）

效果 说明
NoSchedule 如果 Pod 不容忍污点，则不会调度到节点
PreferNoSchedule 软约束，尽量避免但不禁止
NoExecute 已调度的 Pod 如果不容忍污点，会被驱逐

容忍度示例

# 例子1：容忍特定污点
spec:
  tolerations:
  - key: "key"
    operator: "Equal"
    value: "value"
    effect: "NoSchedule"

# 例子2：容忍所有 key=node-role.kubernetes.io/master 的污点
spec:
  tolerations:
  - key: "node-role.kubernetes.io/master"
    operator: "Exists"
    effect: "NoSchedule"

# 例子3：容忍所有污点（不推荐）
spec:
  tolerations:
  - operator: "Exists"

默认污点

污点 节点 说明
node-role.kubernetes.io/master Master/控制平面节点 不允许普通 Pod 调度
node-role.kubernetes.io/control-plane 控制平面节点 不允许普通 Pod 调度
node.kubernetes.io/not-ready NotReady 状态的节点 NoExecute，驱逐现有 Pod
node.kubernetes.io/unreachable 不可达的节点 NoExecute，驱逐现有 Pod
node.kubernetes.io/disk-pressure 磁盘压力 禁止调度新 Pod
node.kubernetes.io/memory-pressure 内存压力 禁止调度新 Pod
node.kubernetes.io/pid-pressure PID 压力 禁止调度新 Pod

拓扑传播约束

拓扑传播约束（Topology Spread Constraints）控制 Pod 在不同拓扑域的分布，如确保高可用。

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1  # 最大差异
        topologyKey: topology.kubernetes.io/zone  # 拓扑域
        whenUnsatisfiable: DoNotSchedule  # 满足策略才调度，或 ScheduleAnyway
        labelSelector:
          matchLabels:
            app: myapp
      containers:
      - name: myapp
        image: myapp

参数说明

参数 说明
maxSkew 不同拓扑域间 Pod 数量的最大差异
topologyKey 拓扑域标签
whenUnsatisfiable 不满足约束时的行为

whenUnsatisfiable

值 说明
DoNotSchedule 不满足约束时不调度
ScheduleAnyway 即使不满足也照常调度

Pod 优先级和抢占

Pod 优先级和抢占（Pod Priority and Preemption）确保重要 Pod 在资源不足时能够运行。

工作原理

  • PriorityClass：定义优先级类
  • 优先级值：数值越高优先级越高
  • 抢占：高优先级 Pod 可以抢占低优先级 Pod 的资源

PriorityClass 示例

# 创建高优先级类
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000  # 优先级值
globalDefault: false  # 是否为默认
description: "High priority class for important applications"
# Pod 使用优先级类
apiVersion: v1
kind: Pod
metadata:
  name: important-pod
spec:
  priorityClassName: high-priority
  containers:
  - name: important
    image: important-image

系统保留优先级

优先级类 优先级值 说明
system-cluster-critical 2000000000 (2*10^9) 关键系统组件
system-node-critical 2000001000 节点关键组件

本章总结

关键要点：
  • Kubernetes 调度器通过预选和优选阶段完成 Pod 调度
  • 节点选择器用于简单的标签匹配
  • 节点亲和性支持硬约束和软约束
  • Pod 亲和性和反亲和性根据已运行的 Pod 调度
  • 污点和容忍度用于控制 Pod 到节点的调度
  • 拓扑传播约束控制 Pod 在拓扑域的分布
  • Pod 优先级和抢占确保重要 Pod 能够运行
  • 可以通过自定义调度器或调度框架扩展调度功能
