🌐 网络

理解服务发现、负载均衡和网络策略

网络概述

Kubernetes 集群的网络模型非常独特，它强调每个 Pod 都有自己的 IP 地址，不需要创建额外的网络地址转换（NAT）。

Kubernetes 网络模型原则

  • 所有 Pod 不运行在 NAT 后面：Pod 可以直接与任何其他 Pod 通信
  • 所有 Node 可以与所有 Pod 通信：不需要额外的网络转换
  • Pod 看到自己看到的 IP：Pod 看到的 IP 就是它们自己的 IP
  • 扁平网络：所有 Pod 属于同一个扁平的网络空间

Service 核心概念

Service 是 Kubernetes 中一种抽象，它定义了一组 Pod 的逻辑集合和访问这些 Pod 的策略。

Service 的作用

  • 服务发现：为一组 Pod 提供稳定的访问端点
  • 负载均衡：将流量分发到后端 Pod
  • 稳定的网络标识：Pod 可能变化，但 Service 地址不变
  • 跨节点访问：允许集群内外访问应用
Service (ClusterIP: 10.96.0.100)
↑ 负载均衡 ↑
Pod 1 (10.244.1.5)
Pod 2 (10.244.2.8)
Pod 3 (10.244.3.2)

Service 类型

类型 访问方式 访问范围 示例场景
ClusterIP 集群内部 IP 仅集群内部 微服务间通信
NodePort 节点 IP + 端口 集群内外 开发测试环境
LoadBalancer 云负载均衡器 公网访问 生产环境公网服务
ExternalName 外部 DNS 名称 外部服务 访问外部数据库等

Service 配置示例

# ClusterIP Service
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: myapp
  ports:
  - protocol: TCP
    port: 80        # Service 端口
    targetPort: 8080  # Pod 目标端口
  type: ClusterIP

# NodePort Service
spec:
  type: NodePort
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080  # 节点端口（30000-32767）

# LoadBalancer Service
spec:
  type: LoadBalancer
  loadBalancerIP: 192.168.1.100  # 可选：指定 IP
  ports:
  - port: 80
    targetPort: 8080

ExternalName Service

apiVersion: v1
kind: Service
metadata:
  name: my-database
spec:
  type: ExternalName
  externalName: database.example.com
  # 访问 my-database.default.svc.cluster.local 会被 DNS 解析为 database.example.com

Headless Service 重要

Headless Service 是一种不分配 ClusterIP 的 Service，通过它可以直接访问单个 Pod。

Headless Service 的特点

  • 无 ClusterIP：不分配集群 IP
  • DNS 直接返回 Pod IP：DNS 查询返回所有后端 Pod 的 IP
  • 用于有状态服务：StatefulSet 必须使用 Headless Service
  • 客户端负载均衡：客户端自己决定访问哪个 Pod
apiVersion: v1
kind: Service
metadata:
  name: my-headless-service
spec:
  clusterIP: None  # 设置为 None 即为 Headless
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080

普通 Service vs Headless Service DNS 查询

类型 DNS 返回 用途
普通 Service Service 的 ClusterIP 负载均衡访问后端
Headless Service 所有后端 Pod 的 IP 直接访问特定 Pod、有状态服务

Ingress 重要

Ingress 是 Kubernetes 中暴露 HTTP/HTTPS 路由规则的 API，它提供从集群外部到集群内部服务的路由。

Ingress vs Service

特性 Service Ingress
协议 仅支持 TCP/UDP 支持 HTTP/HTTPS
路由 基于 IP 和端口 基于主机名和路径
复杂度 简单 高级路由
所需数量 每个服务一个 可以通过一个暴露多个服务

↓ HTTP/HTTPS ↓

Ingress 配置示例

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx  # 指定 Ingress Controller
  tls:
  - hosts:
    - www.example.com
    secretName: my-tls-secret  # TLS 证书
  rules:
  - host: www.example.com
    http:
      paths:
      - path: /app1
        pathType: Prefix
        backend:
          service:
            name: service1
            port:
              number: 80
      - path: /app2
        pathType: Prefix
        backend:
          service:
            name: service2
            port:
              number: 80

Ingress Controller

Ingress 资源本身不执行任何操作，需要部署 Ingress Controller 来实现 Ingress 规则。

Ingress Controller 特点
Nginx Ingress 最常用、功能丰富、性能好
Traefik 自动服务发现、配置简单
Kong API 网关、插件丰富
HAProxy 成熟稳定、负载均衡
Contour 基于 Envoy、HTTP/3 支持
Istio Gateway 服务网格网关

PathType

PathType 说明 示例
Exact 精确匹配路径 /foo 只匹配 /foo
Prefix 前缀匹配路径 /foo 匹配 /foo、/foo/bar
ImplementationSpecific 由控制器决定 取决于具体 Ingress Controller

Gateway API

Gateway API 是 Kubernetes 中一套新一代的 API，用于配置服务网络。

Gateway API vs Ingress

特性 Ingress Gateway API
可扩展性 有限 高（可自定义协议）
角色分离 单角色 多角色（集群运维、应用开发者）
协议支持 HTTP/HTTPS HTTP、gRPC、TCP、UDP 等
成熟度 成熟稳定 发展中（v1.0 已发布）

Gateway API 核心资源

资源 说明
GatewayClass 定义 Gateway 的实现方式和配置
Gateway 定义网络基础设施（LB、IP、端口）
HTTPRoute 定义 HTTP 流量的路由规则
TCPRoute 定义 TCP 流量的路由规则
ReferenceGrant 授权跨越命名空间引用资源

EndpointSlices

EndpointSlices 是 Kubernetes 中用于跟踪 Pod IP 端点的资源，它是 Endpoints 的扩展，提供可扩展和可扩展的端点跟踪。

EndpointSlices 的优势

  • 可扩展：每个 EndpointSlice 最多 100 个端点，大量端点时创建多个
  • 按拓扑分组：可以按节点、区域等拓扑信息分组
  • 减少网络流量：只同步变化的部分
  • 支持多协议：支持 TCP、UDP、SCTP 协议

Endpoints vs EndpointSlices

特性 Endpoints EndpointSlices
端点数量 所有端点在一个对象中 分片存储（最多 100 个/片）
拓扑信息 无 有（zone、hostname）
默认使用 废弃中 默认（v1.21+）

网络策略 安全

网络策略（Network Policy）是 Kubernetes 中用于控制 Pod 之间网络流量的规则，类似于云安全组。

网络策略的作用

  • 隔离 Pod：限制 Pod 的入站和出站流量
  • 白名单控制：只允许特定的流量通过
  • 标签选择：通过标签选择器匹配 Pod
  • 基于端口/协议：可以按端口和协议控制流量

策略类型

  • Ingress：入站流量（进入 Pod）
  • Egress：出站流量（从 Pod 出去）
  • 两者都指定：同时控制入站和出站
  • 都不指定：默认隔离（所有流量被拒绝）

    网络策略示例

    # 允许来自特定命名空间的流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-frontend
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: frontend
    ports:
    - protocol: TCP
      port: 80
    # 允许出站到特定服务的流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-to-database
spec:
  podSelector:
    matchLabels:
      app: myapp
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 5432
    重要：网络策略需要 CNI 插件支持（如 Calico、Cilium、Weave Net 等）。默认的 kube-proxy 不实现网络策略。

DNS

Kubernetes 集群中包含一个 DNS 服务（CoreDNS），用于服务发现和名称解析。

DNS 记录类型

记录类型 格式 解析结果
<>A 记录 my-svc.my-namespace.svc.cluster.local Service ClusterIP（Headless 返回 Pod IPs）
SRV 记录 _my-port-name._my-port-protocol.my-svc.my-namespace.svc.cluster.local 服务端口信息
PTR 记录 Pod IP 反向查询 Pod 主机名

DNS 名称规范

# 完整 FQDN 格式
<service-name>.<namespace>.svc.<cluster-domain>

# 默认集群域：cluster.local
# 示例：
my-service.default.svc.cluster.local

# 简写：
- 同命名空间：my-service
- 不同命名空间：my-service.other-namespace
- 完整：my-service.default.svc.cluster.local

Headless Service DNS

对于 Headless Service，DNS 查询会返回 A 记录指向所有 Pod 的 IP 而不是单个 ClusterIP。

# StatefulSet 示例
# Pod 名称：web-0, web-1, web-2
# DNS 名称：
web-0.web.default.svc.cluster.local  →  10.244.1.5
web-1.web.default.svc.cluster.local  →  10.244.2.6
web-2.web.default.svc.cluster.local  →  10.244.3.7

IPv4/IPv6 双栈

Kubernetes 支持同时使用 IPv4 和 IPv6 地址，实现双栈网络。

双栈网络特性

  • Pod 双地址：每个 Pod 可以同时拥有 IPv4 和 IPv6 地址
  • 服务双地址：Service 可以同时监听 IPv4 和 IPv6
  • 自动选择：客户端自动选择合适的协议族（Happy Eyeballs）

启用双栈网络

# kube-controller-manager 配置
--cluster-cidrs=10.244.0.0/16,fd00::/64

# kube-proxy 配置
--cluster-cidrs=10.244.0.0/16,fd00::/64
--force-ipv6=false

双栈 Service

# 双栈 Service
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  ipFamilies:
  - IPv4
  - IPv6
  ipFamilyPolicy: RequireDualStack
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080

IPFamilyPolicy

策略 说明
RequireDualStack 要求双栈，失败则不创建
PreferDualStack 优先双栈，不可用时使用单栈
SingleStack 单栈，使用主协议族

拓扑感知路由

拓扑感知路由（Topology Aware Hints）是 Kubernetes 的一种优化特性，它使流量尽可能在同一拓扑域（可用区、主机）内路由，以减少延迟和成本。

拓扑感知路由的优势

  • 降低延迟：流量在同一区域内路由
  • 减少成本：减少跨区域流量费用
  • 提高性能：减少网络跳数
Service
可用区 A
Pods
流量优先路由到此
可用区 B
Pods
可用区 C
Pods

启用拓扑感知路由

apiVersion: v1
kind: Service
metadata:
  name: my-service
  annotations:
    service.kubernetes.io/topology-aware-hints: auto
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
工作原理：EndpointSlice 控制器根据节点的拓扑标签（如 `topology.kubernetes.io/zone`）和可用资源，将 Pod 均匀分配到各个拓扑域。

本章总结

关键要点：
  • Kubernetes 网络模型强调扁平 IP 网络，所有 Pod 可直接通信
  • Service 提供稳定的服务发现和负载均衡
  • Service 类型包括 ClusterIP、NodePort、LoadBalancer 和 ExternalName
  • Headless Service 用于有状态服务，DNS 直接返回 Pod IP
  • Ingress 提供 HTTP/HTTPS 路由，需要 Ingress Controller 实现
  • Gateway API 是新一代服务网络 API，功能更强大
  • 网络策略控制 Pod 间流量，需要 CNI 插件支持
  • CoreDNS 提供服务发现，支持 A、SRV、PTR 记录
  • Kubernetes 支持 IPv4/IPv6 双栈网络
  • 拓扑感知路由优化流量路由，降低延迟和成本
