🔧 集群管理

掌握集群运维和监控

集群管理概述

Kubernetes 集群管理涉及节点管理、升级、监控、日志收集、故障排查等多个方面，是保证集群稳定运行的关键。

集群管理的核心职责

  • 节点管理：添加、维护、移除节点
  • 版本升级：集群版本升级和维护
  • 监控告警：监控集群和应用状态
  • 日志管理：收集和分析日志
  • 故障排查：诊断和解决集群问题
  • 备份恢复：etcd 备份和集群恢复

节点管理

Kubernetes 集群由多个节点组成，可以是物理机、虚拟机或云主机。节点管理包括节点的添加、维护、监控和故障处理。

节点状态

状态 说明
Ready 节点健康，可以调度 Pod
NotReady 节点不健康，不能调度 Pod
Unknown 控制平面无法与节点通信
Maintenance 节点处于维护模式（已排空）

节点维护（排空和取消排空）

# 排空节点（驱逐所有 Pod）
kubectl drain node1 --ignore-daemonsets --delete-emptydir-data

# 查看节点排空状态
kubectl get nodes

# 取消排空（恢复节点调度）
kubectl uncordon node1

节点污点

# 添加污点（禁止调度 Pod）
kubectl taint nodes node1 key=value:NoSchedule

# 移除污点
kubectl taint nodes node1 key:NoSchedule-

节点关闭

Kubernetes 支持优雅关闭节点，避免 Pod 突然终止。

# API Server 配置
--pod-eviction-timeout=5m0s  # Pod 驱逐超时时间

集群升级

Kubernetes 支持集群版本升级，可以一次升级一个小版本（如 v1.24 → v1.25）。

升级前准备

  • 备份 etcd 数据
  • 检查兼容性问题
  • 查看变更日志
  • 准备回滚方案

升级步骤

1. 升级第一个控制平面节点
↓
2. 升级其他控制平面节点
↓
3. 升级工作节点
↓
4. 验证集群功能
↓
5. 更新 kubectl 和其他工具
升级注意事项：
  • 不要跨版本升级（如 v1.23 → v1.25）
  • 升级期间 Pod 可能被重新调度
  • API 弃用会导致对象无法创建
  • 充分测试后再升级生产集群

监控

Kubernetes 集群监控包括集群组件监控、节点监控和应用监控，用于及时发现和解决问题。

监控层次

层次 监控内容
基础设施层 节点资源（CPU、内存、磁盘、网络）
集群层 控制平面组件（API Server、Scheduler、Controller Manager
应用层 Pod 和应用的性能指标

常用监控指标

指标类型 示例
资源使用率 CPU%、内存%、磁盘使用率
Pod 状态 运行中、失败、重启次数
节点状态 就绪、不可用、资源压力
集群组件 API Server QPS、Scheduler 调度延迟
网络性能 网络延迟、流量、错误率
存储性能 IOPS、吞吐量、延迟

监控工具

工具 特点
Prometheus 时序数据库，收集和存储指标
Grafana 可视化仪表板和告警
Metrics Server 内置资源监控，用于 HPA
Kube-state-metrics 提供 Kubernetes 对象指标
cAdvisor 容器资源监控
Node Exporter 节点硬件监控

日志

Kubernetes 日志分为系统日志和应用日志，使用日志收集系统统一管理日志。

日志类型

类型 说明
容器日志 容器标准输出和错误输出
系统日志 Kubernetes 组件日志（/var/log）
审计日志 记录 API 请求和响应
事件日志 Kubernetes 事件资源

日志收集架构

Pod/容器
↓ 生成日志 ↓
节点上的日志文件
↓ 采集 ↓
日志收集器（Fluentd/Fluent Bit
↓ 发送 ↓
日志存储（ES/Loki）
↓ 查询 ↓
可视化界面（Kibana/Grafana

常用日志工具

工具 特点
Fluentd 成熟稳定，插件丰富
Fluent Bit 轻量级，资源占用少
Logstash 功能强大，过滤丰富
Filebeat 轻量级，ELK Stack 专用
Loki 轻量级日志存储，与 Grafana 集成

查看日志

# 查看 Pod 日志
kubectl logs <pod-name>

# 实时跟踪日志
kubectl logs -f <pod-name>

# 查看指定容器的日志
kubectl logs <pod-name> -c <container-name>

# 查看上一个容器的日志（容器重启后）
kubectl logs <pod-name> --previous

# 查看所有 Pod 的日志（kubectl 插件）
kubectl logs -l app=myapp --all-containers=true

故障排查

Kubernetes 集群故障排查需要系统化的方法，从基础设施到应用逐步排查。

故障排查流程

1. 确定问题范围（节点/应用）
↓
2. 检查节点状态
↓
3. 检查 Pod 状态
↓
4. 查看事件和日志
↓
5. 检查配置和资源
↓
6. 分析根本原因
↓
7. 修复问题并验证

常见问题

问题 可能原因 排查方法
ImagePullBackOff 镜像拉取失败 检查镜像名称、凭证、仓库可用性
CrashLoopBackOff 容器反复重启 检查日志、健康检查、资源配置
OOMKilled 内存超限 增加内存限制或优化应用
Pod Pending 调度失败 检查资源、调度约束、污点容忍度
节点 NotReady 节点不健康 检查 kubelet、网络、资源
DNS 解析失败 Service/DNS 问题 检查 CoreDNS、Service、网络策略

常用调试命令

命令 说明
kubectl get events 查看集群事件
kubectl describe pod <name> 查看 Pod 详细信息
kubectl logs <pod> 查看 Pod 日志
kubectl top nodes 查看节点资源使用
kubectl top pods 查看 Pod 资源使用
kubectl get svc -o wide 查看 Service 及其 Endpoints

备份和恢复

etcd 是 Kubernetes 的状态数据库，备份 etcd 对于集群安全至关重要。

备份 etcd

# etcdctl 备份
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /path/to/backup.db

恢复 etcd

# 停止静态 Pod 控制平面组件
mv /etc/kubernetes/manifests /etc/kubernetes/manifests.bak

# 恢复 etcd
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  --data-dir=/var/lib/etcd \
  snapshot restore /path/to/backup.db

# 恢复静态 Pod 控制平面组件
mv /etc/kubernetes/manifests.bak /etc/kubernetes/manifests

Velero 备份工具

Velero 是一个 Kubernetes 备份和恢复工具，可以备份整个集群或特定资源。

# 安装 Velero
velero install --provider aws --plugins velero/velero-plugin-for-aws:v1.2.0

# 备份集群
velero backup create my-backup --include-namespaces default

# 恢复集群
velero restore create --from-backup my-backup
