MongoDB 数据库与集合

理解 MongoDB 的数据组织层次：数据库、集合与文档

数据层次结构

MongoDB 使用灵活的层次结构来组织数据。数据从顶层到底层依次为： MongoDB 实例 包含多个 数据库， 每个数据库包含多个 集合， 每个集合包含多个 文档。

层次关系

MongoDB 实例
↓
数据库 (Database)
↓
集合 (Collection)
↓
文档 (Document)
关键概念： MongoDB 的层次结构与传统关系型数据库的 实例 -> 数据库 -> 表 -> 行 结构类似，但更加灵活。集合不强制要求文档具有相同的结构，不同文档可以拥有不同的字段。

数据库操作

切换数据库

使用 use 命令切换到指定数据库：

use myDatabase
提示： 如果数据库不存在，use 命令不会立即创建数据库。数据库会在第一次写入数据时自动创建。

自动创建

MongoDB 在以下操作发生时会自动创建数据库：

  • 向集合中插入第一个文档时
  • 在集合上创建第一个索引时
  • 使用 createCollection 显式创建集合时

查看当前数据库

db

默认连接到 test 数据库。

查看所有数据库

show dbs
注意： 新创建的空数据库不会出现在 show dbs 的结果中，直到其中插入了数据。

删除数据库

use myDatabase
db.dropDatabase()

数据库命名规则与限制

  • 数据库名不区分大小写，但建议全部小写
  • 名称长度最多 64 字节
  • 不能包含以下字符：/ \ . " $ * < > : | ?
  • 不能为空字符串
  • 不能以 $ 开头
  • 不能包含空字符 (\0)

保留数据库名称

保留名称 用途 说明
admin 管理数据库 存储管理员用户、系统级配置信息。如需将用户添加到 admin 数据库，该用户将获得所有数据库的权限
local 本地数据库 存储复制过程中使用的 oplog 和其他本地信息。local 数据库不会被复制到副本集中的其他成员
config 配置数据库 存储分片集群的元数据信息，包括分片信息、块（chunks）分布等
重要： 不要将应用数据存储在 admin、local 或 config 数据库中。这些数据库用于 MongoDB 内部系统功能。

集合操作

隐式创建

MongoDB 在以下情况下会自动创建集合，无需显式创建：

  • 第一次向集合中 insert 文档时
  • 第一次在集合上调用 createIndex 时
// 插入文档时自动创建集合
db.myCollection.insertOne({ name: "张三", age: 28 })

// 创建索引时自动创建集合
db.myCollection.createIndex({ name: 1 })

显式创建

如果需要指定集合选项（如固定大小、验证规则等），可以使用 db.createCollection() 显式创建： 

// 创建普通集合（带文档验证）
db.createCollection("users", {
    validator: {
        $jsonSchema: {
            bsonType: "object",
            required: ["name", "email"],
            properties: {
                name: { bsonType: "string" },
                email: { bsonType: "string", pattern: "^.+@.+$" },
                age: { bsonType: "int", minimum: 0, maximum: 150 }
            }
        }
    },
    validationLevel: "strict",
    validationAction: "error"
})

集合命名规则

  • 集合名不能以系统保留前缀 system. 开头
  • 不能包含 $ 字符（$ 用于运算符）
  • 不能为空字符串
  • 不能包含空字符 (\0)
  • 以 . 开头的集合名仅供内部使用
  • 名称空间 <database>.<collection> 的总长度不能超过 120 字节

查看集合

// 列出当前数据库的所有集合
show collections

// 获取集合的详细信息
db.getCollectionInfos("myCollection")

// 检查集合是否存在
db.getCollectionNames()

删除集合

db.myCollection.drop()

特殊集合类型

MongoDB 提供多种特殊集合类型，针对不同的使用场景进行优化：

类型 特点 使用场景
固定大小集合 (Capped) 固定大小，FIFO 覆盖写入；按插入顺序存储；不支持删除单个文档 日志存储、缓存数据、高吞吐量插入场景
时间序列集合 按时间优化存储和查询；自动管理内部存储结构；支持自动过期 IoT 传感器数据、金融行情数据、监控指标
集群化集合 (Clustered) 按集群键 (Clustered Key) 排序物理存储；减少索引开销；优化范围查询 顺序访问优化场景、需要按特定键排序的查询
视图 (View) 只读虚拟集合；通过聚合管道定义；不占用额外存储空间 数据抽象层、安全隔离、简化复杂查询
按需物化视图 按需计算和缓存结果；可以手动触发刷新；减少重复聚合计算 报表生成、复杂聚合结果缓存
选择建议：
  • 需要高吞吐量日志写入 -> 选择固定大小集合
  • 处理带时间戳的测量数据 -> 选择时间序列集合
  • 需要按特定键顺序物理存储 -> 选择集群化集合
  • 需要数据抽象或安全隔离 -> 选择视图
  • 需要缓存聚合结果 -> 选择按需物化视图

模式验证

灵活模式（默认行为）

MongoDB 默认采用灵活模式（Schema-less），同一集合中的文档可以拥有完全不同的结构。 这种设计使得应用可以随着需求变化而演进，无需执行数据库迁移。 

// 同一集合中的文档可以结构不同
db.products.insertOne({ name: "手机", price: 3999 })
db.products.insertOne({ name: "耳机", price: 299, color: "黑色", brand: "Sony" })

强制模式验证

虽然 MongoDB 默认灵活，但你可以在集合级别定义验证规则，在插入和更新操作时强制执行。 使用 $jsonSchema 可以定义严格的文档结构验证。 

// 创建带验证规则的集合
db.createCollection("students", {
    validator: {
        $jsonSchema: {
            bsonType: "object",
            required: ["name", "studentId", "major"],
            properties: {
                name: {
                    bsonType: "string",
                    description: "必须为字符串类型且为必填字段"
                },
                studentId: {
                    bsonType: "string",
                    pattern: "^STU\\d{6}$",
                    description: "学号格式：STU + 6位数字"
                },
                major: {
                    enum: ["计算机科学", "数学", "物理", "化学", "生物"],
                    description: "专业必须是列表中的值之一"
                },
                gpa: {
                    bsonType: "double",
                    minimum: 0,
                    maximum: 4.0,
                    description: "GPA 范围 0-4.0"
                },
                enrollDate: {
                    bsonType: "date",
                    description: "入学日期"
                }
            }
        }
    },
    validationLevel: "strict",
    validationAction: "error"
})

验证级别

级别 说明
strict 对所有插入和更新操作都执行验证（默认值）
moderate 仅对已符合验证规则的文档执行更新验证；不符合规则的历史文档更新时不验证
off 不执行任何验证

验证动作

动作 说明
error 验证失败时拒绝操作并返回错误（默认值）
warn 验证失败时记录警告但允许操作继续

修改验证规则

// 添加验证规则到已有集合
db.runCommand({
    collMod: "students",
    validator: {
        $jsonSchema: {
            bsonType: "object",
            required: ["name", "studentId"],
            properties: {
                name: { bsonType: "string" },
                studentId: { bsonType: "string" }
            }
        }
    },
    validationLevel: "moderate"
})
最佳实践： 在开发阶段可以使用 validationAction: "warn" 来识别不合规数据而不阻塞操作，确认验证规则无误后再切换为 "error"。

集合 UUID

每个集合在创建时都会被分配一个不可变的 UUID（通用唯一标识符）。这个 UUID 在整个副本集和分片集群中保持一致。

获取集合 UUID

// 方法一：通过 listCollections
db.runCommand({ listCollections: 1 })

// 方法二：通过 getCollectionInfos
db.getCollectionInfos({ name: "myCollection" })

返回结果示例：

{
    "name": "myCollection",
    "type": "collection",
    "options": {},
    "info": {
        "readOnly": false,
        "uuid": new UUID("a5c5f1d2-8e4b-4d3a-9f7e-6b2c1d8e4a3f")
    },
    "idIndex": { ... }
}
UUID 的作用：
  • 不可变性： 集合的 UUID 在创建后无法更改，即使集合被重命名
  • 一致性： 在副本集中，同一集合在主节点和所有从节点上具有相同的 UUID
  • 分片集群： 集群中所有分片上的同一集合共享相同的 UUID
  • 用途： UUID 用于内部操作和故障恢复，确保跨节点数据一致性

固定大小集合详解

创建固定大小集合

db.createCollection("log", {
    capped: true,
    size: 100000,    // 字节为单位，集合最大大小为 100KB
    max: 5000        // 最多存储 5000 条文档
})
参数说明： size 是必填参数，max 是可选的。当同时指定时，先达到哪个限制就触发覆盖。即使设置了 max，也必须指定 size，因为 MongoDB 需要预分配存储空间。

FIFO 行为

固定大小集合采用先进先出（FIFO）策略：当集合达到大小上限时，新插入的文档会覆盖最早的文档。

新文档插入
↓
检查是否超出大小限制
↓
超出限制：删除最早的文档
↓
插入新文档

固定大小集合的限制

  • 不能删除单个文档： 只能通过插入新文档自动覆盖，或使用 db.collection.drop() 删除整个集合
  • 自然顺序即插入顺序： 文档按插入顺序物理存储，自然排序（$natural）等于插入顺序
  • 不能添加二级索引： 固定大小集合不支持创建额外的索引（除默认的 _id 索引）
  • 不能分片： 固定大小集合不支持分片
  • 更新操作限制： 更新文档时，文档大小不能增加，否则操作失败

查询固定大小集合

// 按插入顺序查询（默认）
db.log.find()

// 按插入逆序查询（最新文档在前）
db.log.find().sort({ $natural: -1 })

// 使用 tailable cursor（类似 tail -f）
db.log.find({}, { tailable: true })

检查集合是否为固定大小集合

db.log.isCapped()   // 返回 true 或 false

将普通集合转换为固定大小集合

db.runCommand({
    convertToCapped: "myCollection",
    size: 100000,
    max: 5000
})
使用场景示例： 固定大小集合非常适合存储系统日志、审计记录、缓存热点数据等场景，因为这些数据通常只需保留最近的内容，且写入频率很高。

MongoDB vs SQL 术语对比

MongoDB 与传统关系型数据库在概念上存在对应关系，但实现方式差异很大。下表帮助你快速理解两种数据库之间的术语映射：

MongoDB 概念 SQL 概念 说明
数据库 (Database) 数据库 (Database) 两者概念一致，都是数据组织的顶层容器
集合 (Collection) 表 (Table) 集合不强制模式，表需要预定义列结构
文档 (Document) 行 (Row) 文档是 BSON 格式，可嵌套；行是扁平结构
字段 (Field) 列 (Column) 同一集合的文档可以有不同字段；表的行共享列定义
嵌入文档 JOIN 关联 MongoDB 倾向于将关联数据嵌入文档中；SQL 通过 JOIN 实现跨表查询
_id 字段 主键 (Primary Key) MongoDB 自动生成 ObjectId；SQL 需手动定义主键

↔
Database

↔

↔

核心差异： SQL 数据库要求严格的表结构定义（DDL），任何结构变更都需要 ALTER TABLE 操作。MongoDB 的灵活模式允许文档自由演进，不同文档可以有不同的字段，非常适合快速迭代的应用开发。
