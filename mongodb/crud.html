CRUD 操作

创建 / 读取 / 更新 / 删除 — MongoDB 数据操作核心

CRUD 概述

CRUD 是 Create（创建）、Read（读取）、Update（更新）、Delete（删除）四种数据操作的缩写，是所有数据库系统的核心功能。MongoDB 提供了丰富且灵活的 CRUD 操作 API，支持对文档的增删改查。

Create

插入文档

insertOne() insertMany()

Read

查询文档

find() findOne()

Update

更新文档

updateOne() updateMany() replaceOne()

Delete

删除文档

deleteOne() deleteMany()
核心原则：MongoDB 中所有写操作（Create、Update、Delete）在单个文档级别上是原子操作（Atomic）。这意味着对单个文档的修改要么全部成功，要么全部失败，不会出现部分更新的情况。

创建操作 (Create)

创建操作将新文档插入到集合中。如果插入操作时集合不存在，MongoDB 会自动创建该集合。

db.collection.insertOne()

向集合中插入单个文档。返回包含 insertedId 的结果。

示例：插入单个文档 
db.users.insertOne({
  name: "张三",
  age: 28,
  email: "zhangsan@example.com",
  hobbies: ["阅读", "编程"],
  address: {
    city: "北京",
    district: "海淀区"
  }
})

// 返回结果
{ "acknowledged": true, "insertedId": ObjectId("6502...") }

db.collection.insertMany()

向集合中插入多个文档。接受文档数组作为参数，可指定 ordered 选项。

示例：插入多个文档 
db.users.insertMany([
  { name: "李四", age: 32, email: "lisi@example.com" },
  { name: "王五", age: 25, email: "wangwu@example.com" },
  { name: "赵六", age: 29, email: "zhaoliu@example.com" }
], { ordered: true })

// 返回结果
{
  "acknowledged": true,
  "insertedIds": [
    ObjectId("6502a1..."),
    ObjectId("6502a2..."),
    ObjectId("6502a3...")
  ]
}
ordered 选项说明：
  • ordered: true（默认）：按顺序执行，遇到错误时停止后续插入
  • ordered: false：无序执行，即使某个文档插入失败也继续插入其余文档

关键特性

  • 自动创建集合：如果集合不存在，插入操作会自动创建集合
  • 自动生成 _id：如果未指定 _id 字段，MongoDB 自动生成 ObjectId
  • 原子性：单个文档的插入操作是原子的
  • _id 唯一性：集合中每个文档的 _id 必须唯一，重复插入会报错
注意：_id 字段在文档插入后不可修改。如果手动指定 _id，请确保其唯一性，否则会抛出 duplicate key error。

读取操作 (Read)

读取操作从集合中检索文档。MongoDB 使用 db.collection.find() 方法查询文档，支持丰富的查询操作符和投影操作。

基本查询

示例：基本查询 
// 查询所有文档
db.users.find({})

// 等值查询
db.users.find({ age: 25 })

// 查询单个文档
db.users.findOne({ email: "zhangsan@example.com" })

比较操作符

比较操作符用于对字段值进行比较筛选。

操作符 说明 示例
$eq 等于 { age: { $eq: 25 } } 等同于 { age: 25 }
$gt / $gte 大于 / 大于等于 { age: { $gt: 20 } }
$lt / $lte 小于 / 小于等于 { age: { $lt: 30 } }
$in 在数组中（匹配任一值） { age: { $in: [20, 25, 30] } }
$ne 不等于 { age: { $ne: 25 } }
$nin 不在数组中 { age: { $nin: [20, 25] } }
示例：范围查询 
// 查询年龄在 20 到 30 之间的用户
db.users.find({
  age: { $gte: 20, $lte: 30 }
})

逻辑操作符

操作符 说明 示例
$and 逻辑与，所有条件同时满足 { $and: [{ age: { $gt: 20 } }, { city: "北京" }] }
$or 逻辑或，满足任一条件 { $or: [{ age: { $lt: 20 } }, { age: { $gt: 60 } }] }
$not 逻辑非，取反操作 { age: { $not: { $gt: 30 } } }
$nor 所有条件都不满足 { $nor: [{ age: 20 }, { age: 25 }] }
隐式 $and：当同一字段使用多个操作符时，MongoDB 隐式使用 $and。例如 { age: { $gt: 20, $lt: 30 } } 等同于 { $and: [{ age: { $gt: 20 } }, { age: { $lt: 30 } }] }。

元素操作符

操作符 说明 示例
$exists 判断字段是否存在 { phone: { $exists: true } }
$type 判断字段类型 { age: { $type: "int" } }
示例：元素操作符 
// 查询包含 phone 字段的文档
db.users.find({ phone: { $exists: true } })

// 查询 age 字段类型为 string 的文档（数据类型不一致问题排查）
db.users.find({ age: { $type: "string" } })

数组操作符

操作符 说明 示例
$all 匹配包含所有指定元素的数组 { tags: { $all: ["MongoDB", "数据库"] } }
$elemMatch 匹配数组中满足所有条件的元素 { results: { $elemMatch: { $gt: 80, $lt: 95 } } }
$size 匹配指定长度的数组 { tags: { $size: 3 } }
示例：数组操作符 
// 查询 hobbies 同时包含 "阅读" 和 "编程" 的文档
db.users.find({
  hobbies: { $all: ["阅读", "编程"] }
})

// 查询 scores 数组中存在同时 > 80 且 < 95 的元素
db.students.find({
  scores: { $elemMatch: { $gt: 80, $lt: 95 } }
})

// 查询恰好有 3 个标签的文档
db.articles.find({
  tags: { $size: 3 }
})

嵌套文档查询（点号表示法）

使用点号表示法可以查询嵌套文档中的字段。点号表示法是 MongoDB 查询嵌入文档的核心机制。

示例：嵌套文档查询 
// 查询 address.city 为 "北京" 的文档
db.users.find({
  "address.city": "北京"
})

// 多层嵌套查询
db.users.find({
  "address.city": "北京",
  "address.district": "海淀区"
})

// 嵌套数组中的对象查询
db.schools.find({
  "students.name": "张三"
})

// 使用 $elemMatch 精确匹配数组内嵌套文档的多个字段
db.schools.find({
  students: {
    $elemMatch: { name: "张三", age: 15 }
  }
})
注意：查询嵌套文档时，使用点号表示法而非直接匹配整个嵌套对象。直接匹配对象要求精确匹配所有字段及顺序，通常不是预期行为。

投影 (Projection)

投影用于控制查询结果中返回的字段，减少网络传输和提高性能。

示例：投影操作 
// 只返回 name 和 email 字段（包含模式）
db.users.find(
  { age: { $gt: 20 } },
  { name: 1, email: 1 }
)
// 结果: { _id: ..., name: "...", email: "..." }

// 排除 address 字段（排除模式）
db.users.find(
  {},
  { address: 0 }
)

// 排除 _id 字段
db.users.find(
  {},
  { _id: 0, name: 1, email: 1 }
)

// 数组投影：只返回数组前 N 个元素
db.articles.find(
  {},
  { comments: { $slice: 5 } }
)

// 数组投影：跳过前 2 个，返回接下来 3 个
db.articles.find(
  {},
  { comments: { $slice: [2, 3] } }
)
投影规则：
  • 1 或 true — 包含该字段
  • 0 或 false — 排除该字段
  • 不能混合使用包含和排除（_id 字段除外，始终可以排除）
  • 默认情况下，_id 字段始终返回，需显式设为 0 才能排除

更新操作 (Update)

更新操作修改集合中已有文档。MongoDB 提供了三种更新方法和丰富的更新操作符。

更新方法

方法 说明
db.collection.updateOne() 更新匹配筛选条件的第一个文档
db.collection.updateMany() 更新匹配筛选条件的所有文档
db.collection.replaceOne() 替换匹配筛选条件的第一个文档（整体替换，保留 _id）

更新操作符

更新操作符是 MongoDB 更新的核心，可以精确控制文档的修改方式。

操作符 说明 示例
$set 设置字段值（字段不存在则创建） { $set: { status: "active" } }
$unset 删除字段 { $unset: { tempField: "" } }
$inc 增加数值（可为负数） { $inc: { loginCount: 1 } }
$mul 乘以数值 { $mul: { price: 1.1 } }
$rename 重命名字段 { $rename: { "oldName": "newName" } }
$push 向数组添加元素 { $push: { tags: "new" } }
$pull 从数组删除匹配元素 { $pull: { tags: "old" } }
$addToSet 添加唯一元素到数组（去重） { $addToSet: { tags: "unique" } }
$pop 删除数组首(=-1)或末(=1)元素 { $pop: { tags: 1 } }
示例：updateOne 使用更新操作符 
// 使用 $set 更新字段
db.users.updateOne(
  { name: "张三" },
  { $set: { age: 29, "address.city": "上海" } }
)

// 使用 $inc 增加数值
db.products.updateOne(
  { _id: 100 },
  { $inc: { stock: -1, soldCount: 1 } }
)

// 使用 $push 和 $addToSet 操作数组
db.users.updateOne(
  { name: "张三" },
  {
    $push: { hobbies: "旅行" },
    $addToSet: { tags: "vip" }
  }
)
示例：updateMany 批量更新 
// 将所有北京用户的 status 更新为 "active"
db.users.updateMany(
  { "address.city": "北京" },
  { $set: { status: "active" } }
)

// 返回结果
{
  "acknowledged": true,
  "matchedCount": 15,
  "modifiedCount": 12
}
示例：replaceOne 整体替换 
// 替换整个文档（_id 保持不变）
db.users.replaceOne(
  { name: "张三" },
  {
    name: "张三",
    age: 30,
    email: "zhangsan@new.com",
    city: "上海"
  }
)

Upsert 选项

upsert: true 选项在找不到匹配文档时自动创建新文档，实现了"更新或插入"的语义。

示例：upsert 操作 
db.users.updateOne(
  { email: "newuser@example.com" },
  { $set: { name: "新用户", age: 20 } },
  { upsert: true }
)

// 如果匹配到文档 → 更新
// 如果未匹配到文档 → 插入新文档：
// { _id: ObjectId("..."), email: "newuser@example.com", name: "新用户", age: 20 }
原子性保证：所有更新操作在单个文档级别是原子的。对单个文档的多个字段修改（如同时 $set 和 $inc）要么全部生效，要么全部不生效。

删除操作 (Delete)

删除操作从集合中移除文档。与更新类似，MongoDB 提供了两种删除方法。

方法 说明 示例
db.collection.deleteOne() 删除匹配筛选条件的第一个文档 db.users.deleteOne({ status: "inactive" })
db.collection.deleteMany() 删除匹配筛选条件的所有文档 db.users.deleteMany({ status: "inactive" })
示例：删除操作 
// 删除单个文档
db.users.deleteOne({ _id: ObjectId("6502a1...") })

// 批量删除：删除所有 status 为 "inactive" 的文档
db.users.deleteMany({ status: "inactive" })

// 删除所有文档（慎用！）
db.users.deleteMany({})

// 返回结果
{ "acknowledged": true, "deletedCount": 15 }
警告：
  • deleteMany({}) 会删除集合中所有文档，但不会删除集合本身
  • 如需删除集合，请使用 db.collection.drop()
  • 删除操作不可逆，请在生产环境中谨慎操作，建议先使用 find() 确认筛选条件

批量写入操作 (Bulk Write)

db.collection.bulkWrite() 允许在单次请求中执行多种类型的写操作，显著提高批量数据处理的效率。

支持的操作类型

insertOne
插入单个文档
updateOne
更新单个文档
updateMany
批量更新文档
deleteOne
删除单个文档
deleteMany
批量删除文档
replaceOne
替换单个文档
示例：bulkWrite 批量操作 
db.users.bulkWrite([
  {
    insertOne: {
      document: { name: "新用户A", age: 22 }
    }
  },
  {
    updateOne: {
      filter: { name: "张三" },
      update: { $set: { status: "active" } },
      upsert: true
    }
  },
  {
    deleteOne: {
      filter: { status: "expired" }
    }
  },
  {
    updateMany: {
      filter: { "address.city": "北京" },
      update: { $inc: { loginCount: 1 } }
    }
  }
], { ordered: true })

有序 vs 无序执行

Ordered（有序执行）

  • 默认模式：ordered: true
  • 按数组顺序逐一执行
  • 遇到错误立即停止后续操作
  • 适用于操作之间有依赖关系的场景
  • 性能相对较低（串行执行）

Unordered（无序执行）

  • 设置：ordered: false
  • MongoDB 可并行执行操作
  • 遇到错误继续执行其余操作
  • 适用于操作之间无依赖关系的场景
  • 性能更高（并行执行）

SQL 到 MongoDB 映射

对于熟悉关系型数据库的开发者，下表展示了常见 SQL 操作与 MongoDB 操作的对应关系。

SQL 操作 MongoDB 操作 说明
INSERT INTO users VALUES(...) db.users.insertOne({...}) 插入单行/文档
INSERT INTO users VALUES(...), (...) db.users.insertMany([...]) 批量插入
SELECT * FROM users db.users.find({}) 查询所有
SELECT * FROM users WHERE age = 25 db.users.find({ age: 25 }) 条件查询
SELECT name, email FROM users db.users.find({}, { name: 1, email: 1 }) 投影查询
SELECT * FROM users WHERE age > 25 db.users.find({ age: { $gt: 25 } }) 范围查询
SELECT * FROM users WHERE age IN (20, 25) db.users.find({ age: { $in: [20, 25] } }) IN 查询
SELECT * FROM users WHERE a=1 OR b=2 db.users.find({ $or: [{ a: 1 }, { b: 2 }] }) OR 查询
SELECT * FROM users ORDER BY age DESC db.users.find({}).sort({ age: -1 }) 排序
SELECT * FROM users LIMIT 10 SKIP 20 db.users.find({}).skip(20).limit(10) 分页
SELECT COUNT(*) FROM users db.users.countDocuments({}) 计数
UPDATE users SET status='A' WHERE age > 30 db.users.updateMany({ age: { $gt: 30 } }, { $set: { status: "A" } }) 批量更新
DELETE FROM users WHERE status='D' db.users.deleteMany({ status: "D" }) 批量删除
SELECT DISTINCT city FROM users db.users.distinct("city") 去重
SELECT * FROM users JOIN orders ON ... db.users.aggregate([{ $lookup: ... }]) 关联查询
术语映射：
  • 表 (Table) → 集合 (Collection)
  • 行 (Row) → 文档 (Document)
  • 列 (Column) → 字段 (Field)
  • 主键 (Primary Key) → _id 字段
  • JOIN → $lookup 聚合阶段

读写关注 (Read/Write Concern)

读写关注（Concern）决定了 MongoDB 在返回操作结果前需要满足的确认级别，是数据一致性与性能之间的关键权衡。

Read Concern（读关注）

读关注级别控制从查询返回的数据的一致性和隔离性程度。

local

默认级别。返回实例的最新数据，不保证该数据已写入大多数节点。可能回滚。

db.users.find().readConcern("local")

available

返回实例的最新数据，与 local 类似但对分片集合查询不会产生碎片。适用于读操作不需要因果一致性的场景。

majority

返回已被大多数副本集成员确认的数据。保证数据不会被回滚（读取已提交）。

db.users.find().readConcern("majority")

linearizable

最强读一致性保证。确保读操作返回的数据在读操作开始前已被大多数节点确认，且读操作期间数据不会变化。适用于对一致性要求极高的场景。

snapshot

仅在多文档事务中可用。读取事务开始时的一致性快照，保证事务内读取的数据一致性。

读关注级别对比：
local / available：性能最优，一致性最弱
majority：平衡性能与一致性，最常用
linearizable / snapshot：一致性最强，性能开销最大

Write Concern（写关注）

写关注决定了 MongoDB 在确认写操作成功前需要多少个节点确认该写入。

w

确认写入的节点数

w: 1 — 主节点确认
w: "majority" — 大多数节点确认
w: 0 — 不确认（fire-and-forget）

j

是否写入日志 (journal)

j: true — 写入持久化到磁盘日志
j: false — 不等待日志写入（默认）

wtimeout

写操作超时时间

超时后写操作仅在指定节点确认
单位：毫秒
wtimeout: 5000

示例：写关注配置 
// 等待大多数节点确认 + 写入日志
db.users.insertOne(
  { name: "张三", age: 28 },
  {
    writeConcern: {
      w: "majority",
      j: true,
      wtimeout: 5000
    }
  }
)

// 仅主节点确认（默认行为）
db.users.insertOne(
  { name: "李四", age: 25 },
  { writeConcern: { w: 1 } }
)

// 不等待确认（最高性能，最低可靠性）
db.logs.insertOne(
  { level: "info", message: "..." },
  { writeConcern: { w: 0 } }
)
最佳实践：
  • 关键业务数据：使用 w: "majority" + j: true 确保数据安全
  • 日志等非关键数据：使用 w: 1 或 w: 0 提升写入性能
  • 始终设置 wtimeout 防止无限等待

查询处理流程

MongoDB 查询从客户端发出到返回结果，经历一系列处理步骤。理解这个流程有助于编写高效查询和排查性能问题。

客户端
Client
➜
mongos / mongod
路由/节点
➜
索引查找
Index Lookup
➜
集合扫描
Collection Scan
➜
返回结果
Result

流程详解

阶段 说明 性能关注点
1. 客户端发送 驱动程序将查询序列化为 BSON，发送到 mongos（分片集群）或 mongod（单节点/副本集） 减少查询字段（使用投影）、限制返回数量（limit）
2. mongos/mongod 接收 解析查询语句，生成查询计划（Query Plan），决定使用哪个索引或全表扫描 查询计划缓存、索引选择
3. 索引查找 如果存在合适的索引，使用 B-tree 索引快速定位文档；否则跳过此步骤 确保查询使用索引（通过 explain() 验证）
4. 集合扫描 索引定位后检查文档（FETCH 阶段），验证是否完全匹配查询条件；无索引时执行全表扫描（COLLSCAN） 避免全表扫描、使用覆盖查询（Covered Query）
5. 返回结果 将匹配的文档序列化为 BSON 返回客户端；支持游标（Cursor）分批返回 合理使用游标、控制批量大小
示例：使用 explain() 分析查询计划 
// 查看查询执行计划
db.users.find({ age: { $gt: 25 } }).explain("executionStats")

// 关键输出字段：
// - winningPlan: 选中的执行计划
// - stage: COLLSCAN（全表扫描） vs IXSCAN（索引扫描）
// - totalDocsExamined: 扫描的文档数
// - totalKeysExamined: 扫描的索引键数
// - executionTimeMillis: 执行耗时

// 理想比例：totalDocsExamined ≈ nReturned（扫描文档数 ≈ 返回文档数）
性能优化建议：
  • 始终为高频查询创建合适的索引
  • 使用 explain() 验证查询是否使用了索引
  • 目标是 COLLSCAN 阶段不出现在生产查询中
  • 覆盖查询（Covered Query）可以跳过 FETCH 阶段，直接从索引返回数据
  • 使用投影减少返回字段，降低网络传输和内存占用
