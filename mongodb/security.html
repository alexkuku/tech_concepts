安全分层架构

从外到内，MongoDB 安全架构呈层层防护结构：

网络层
IP 访问列表 / 网络对等连接 / 私有端点
防火墙与网络隔离
↓
传输层
TLS/SSL 加密通信
防止窃听和中间人攻击
↓
身份认证
SCRAM / x.509 / Kerberos / LDAP / OIDC
验证用户身份
↓
权限控制
RBAC (基于角色的访问控制)
限制操作范围
↓
数据层
静态加密 / 可查询加密 / CSFLE
保护存储数据安全
原则：安全防护应采用纵深防御策略，不能仅依赖单一安全层。每一层都是下一层的保护前提，缺失任何一层都会增加安全风险。

身份认证 (Authentication)

MongoDB 支持多种身份认证机制，适用于不同场景和部署环境：

认证机制 适用版本 适用场景 版本要求
SCRAM 所有版本 默认认证机制，基于用户名/密码 社区版 / 企业版
x.509 所有版本 内部副本集/分片认证、客户端证书认证 社区版 / 企业版
Kerberos 企业功能 与现有 Kerberos 基础设施集成 Enterprise
LDAP 企业功能 与 Active Directory / LDAP 集成 Enterprise
OIDC 7.0+ 与身份提供商（Okta/Azure AD 等）集成 社区版 / 企业版

SCRAM 认证配置

// 创建管理员用户
db.createUser({
  user: "admin",
  pwd: passwordPrompt(),
  roles: [
    { role: "userAdminAnyDatabase", db: "admin" },
    { role: "readWriteAnyDatabase", db: "admin" }
  ]
})

// 启用认证（mongod.conf）
security:
  authorization: enabled

// 连接时认证
mongo --host localhost -u admin -p --authenticationDatabase admin

x.509 证书认证配置

// mongod.conf - 启用 x.509 内部认证
security:
  clusterAuthMode: x509
net:
  tls:
    mode: requireTLS
    certificateKeyFile: /path/to/server.pem
    CAFile: /path/to/ca.pem

// 创建 x.509 用户
db.getSiblingDB("$external").runCommand({
  createUser: "CN=user,OU=dept,O=org,L=city,ST=state,C=CN",
  roles: [{ role: "readWrite", db: "mydb" }]
})
最佳实践：在生产环境中，推荐使用 x.509 证书进行副本集成员间的内部认证，同时使用 SCRAM 或 LDAP 进行客户端认证。

权限控制 (Authorization)

MongoDB 采用基于角色的访问控制（RBAC）。用户被分配角色，角色定义了可执行的操作和可访问的资源。

用户
↓ 分配
角色 (Role)
↓ 定义
权限 (Privilege)
↓ 作用于
资源 (Resource)

内置角色分类

角色类别 角色名称 权限范围
数据库用户 read 读取指定数据库
readWrite 读写指定数据库
dbAdmin 数据库管理（索引、统计等），不含读写
集群管理 clusterAdmin 集群级别的管理操作
clusterMonitor 只读的集群监控权限
hostManager 管理服务器级别操作
备份恢复 backup 备份数据权限
restore 恢复数据权限
跨数据库 readAnyDatabase 读取所有数据库
readWriteAnyDatabase 读写所有数据库
超级用户 root 所有权限（仅限 admin 数据库）

自定义角色示例

// 创建自定义角色
db.createRole({
  role: "orderManager",
  privileges: [
    {
      resource: { db: "mydb", collection: "orders" },
      actions: ["find", "insert", "update"]
    },
    {
      resource: { db: "mydb", collection: "system.profile" },
      actions: ["find"]
    }
  ],
  roles: ["read"]
})

// 为用户分配自定义角色
db.createUser({
  user: "orderUser",
  pwd: passwordPrompt(),
  roles: [{ role: "orderManager", db: "admin" }]
})
最小权限原则：应为每个用户/应用分配完成其工作所需的最小权限集。避免使用 root 或 readWriteAnyDatabase 等宽泛角色。

加密机制 (Encryption)

MongoDB 提供多层次的加密方案，保护数据在传输中和存储时的安全：

1. 静态加密 (Encryption at Rest)

对磁盘上的数据文件进行加密，防止物理存储介质被盗取后的数据泄露。

// mongod.conf - 启用静态加密
security:
  enableEncryption: true
  encryptionKeyFile: /path/to/keyfile
  kmip:
    serverName: kmip.example.com
    port: 5696
    clientCertificateFile: /path/to/client.pem
    serverCAFile: /path/to/ca.pem
特性 说明
加密范围 数据文件、日志文件、oplog 等所有磁盘数据
密钥管理 本地密钥文件或 KMIP 服务器
加密算法 AES-256-CBC（默认）
Atlas 支持 默认启用静态加密，使用云提供商的 KMS

2. 传输加密 (Encryption in Transit)

通过 TLS/SSL 加密客户端与服务器之间、副本集成员之间的网络通信。

// mongod.conf - 启用 TLS
net:
  tls:
    mode: requireTLS
    certificateKeyFile: /path/to/mongodb.pem
    CAFile: /path/to/ca.pem
    allowConnectionsWithoutCertificates: false

3. 可查询加密 (Queryable Encryption)

允许对加密数据直接执行查询，无需解密。客户端自动加密，服务端存储密文。

应用程序
自动加密 / 解密
↓ 密文传输
MongoDB 服务端
存储密文，支持密文查询
↓ 密钥管理
KMS (密钥管理服务)
AWS KMS / Azure Key Vault / GCP KMS

4. 客户端字段级加密 (CSFLE)

在客户端对文档的特定字段进行加密，服务端仅存储密文。支持自动和手动两种加密模式。

// Node.js CSFLE 示例
const { ClientEncryption } = require("mongodb-client-encryption");

const encryption = new ClientEncryption(mongoClient, {
  keyVaultNamespace: "encryption.__keyVault",
  kmsProviders: {
    aws: {
      accessKeyId: process.env.AWS_ACCESS_KEY,
      secretAccessKey: process.env.AWS_SECRET_KEY
    }
  }
});

// 创建数据密钥
const dataKeyId = await encryption.createDataKey("aws", {
  masterKey: { region: "us-east-1", key: "cmk-id" }
});

// 自动加密配置
const autoEncryptionOpts = {
  keyVaultNamespace: "encryption.__keyVault",
  kmsProviders: { ... },
  schemaMap: {
    "mydb.users": {
      bsonType: "object",
      properties: {
        ssn: {
          encrypt: {
            bsonType: "string",
            algorithm: "AEAD_AES_256_CBC_HMAC_SHA_512-Deterministic",
            keyId: [dataKeyId]
          }
        }
      }
    }
  }
};

加密方案对比

加密方案 加密位置 加密粒度 可查询性 版本要求
静态加密 服务端 整个数据库 透明（自动解密） 3.2+ 企业版
传输加密 网络层 所有通信 透明 3.0+
可查询加密 客户端 字段级 支持密文查询 7.0+
CSFLE 客户端 字段级 有限支持 4.2+ 企业版

审计 (Auditing)

MongoDB 企业版和 Atlas 提供审计功能，记录数据库操作以满足合规性要求。

// mongod.conf - 启用审计
auditLog:
  destination: file
  format: JSON
  path: /var/log/mongodb/audit.json
  filter: '{ "atype": { "$in": ["authCheck", "createUser", "dropUser"] } }'
审计目标 说明
文件 输出到 JSON 或 BSON 文件
控制台 输出到 stdout（仅用于调试）
Syslog 发送到系统日志服务
Atlas 审计：MongoDB Atlas 默认记录所有数据库操作和集群事件，可在 Atlas 控制台中查看和导出审计日志，无需额外配置。

Atlas 安全特性

MongoDB Atlas 提供了额外的云安全功能：

安全特性 说明 默认状态
静态加密 所有集群默认启用静态加密，使用云提供商 KMS 默认启用
传输加密 所有连接默认要求 TLS 默认启用
IP 访问列表 限制可连接的 IP 地址范围 需配置
网络对等连接 通过 VPC 对等连接将 Atlas 与应用 VPC 连接 需配置
私有端点 通过 PrivateLink / Private Service Connect 建立私有连接 需配置
数据库审计 自动记录所有数据库操作和集群事件 默认启用
云备份加密 备份快照使用 AES-256 加密 默认启用
建议：在 Atlas 中始终配置 IP 访问列表和网络对等连接，限制只有授权网络能访问数据库，减少攻击面。
