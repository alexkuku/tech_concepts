MongoDB 数据建模

Schema Design Patterns | 嵌入 vs 引用 | 模式验证

1. 核心原则

“

一起访问的数据应该一起存储

Data accessed together should be stored together

”

MongoDB 的数据建模与关系型数据库有本质区别。关系型数据库强调范式化（Normalization），通过外键关联分散存储数据；而 MongoDB 倡导根据应用的数据访问模式来组织数据结构。

关键思路：在 MongoDB 中，数据建模的首要考虑是应用如何使用数据，而非数据本身的结构。如果某些数据总是被一起查询，就应该将它们放在同一个文档中。

2. 灵活模式的优势

MongoDB 的文档模型天然支持灵活模式（Flexible Schema），即同一集合中的文档可以拥有不同的字段结构。这带来了以下核心优势：

📊

字段变化 (Field Variation)

不同文档可以包含不同的字段。例如，不同类型的商品拥有不同的属性字段，无需为每种类型创建单独的表。

📦

类型变化 (Type Variation)

同一字段在不同文档中可以存储不同类型的数据。例如，tags 字段在某个文档中是字符串，在另一个文档中可以是数组。

🔄

迭代式模型改进 (Iterative Improvement)

无需停机即可演进数据模型。新增字段不影响现有文档，应用可以逐步迁移到新模式。

🔗

直接对象映射 (Direct Object Mapping)

BSON 文档天然映射到应用层的对象结构，消除 ORM 层的阻抗失配问题，开发更高效。

✅

按需验证 (On-demand Validation)

通过 $jsonSchema 可在需要时对集合施加验证规则，灵活性与一致性兼得。

提示：灵活模式不等于没有模式。最佳实践是：开发初期利用灵活性快速迭代，生产环境稳定后通过 Schema Validation 保障数据质量。

3. 关系建模

MongoDB 中建模实体间关系有三种基本类型，每种类型有不同的推荐策略：

一对一 (One-to-One)

嵌入（推荐） 推荐嵌入

例如：患者 → 病历。每个患者只有一份病历，两者总是一起访问，将病历嵌入患者文档可一次查询获取所有信息。

// 嵌入方式
{
  "_id": ObjectId("..."),
  "name": "张三",
  "record": {
    "bloodType": "A",
    "allergies": ["penicillin"],
    "diagnoses": [...]
  }
}
一对多 (One-to-Many)

嵌入或引用 视情况而定

例如：用户 → 帖子。如果"多"的一端数据量少且总是一起访问，适合嵌入；如果数据量大或需要独立访问，适合引用。

// 少量数据：嵌入
{ "name": "李四", "posts": [
    { "title": "帖1", "content": "..." },
    { "title": "帖2", "content": "..." }
]}

// 大量数据：引用
{ "name": "李四", "postIds": [ObjectId("..."), ObjectId("...")] }
// 或在帖子中引用用户
{ "title": "帖1", "authorId": ObjectId("..."), "content": "..." }
多对多 (Many-to-Many)

引用 推荐引用

例如：学生 ↔ 课程。一个学生可选多门课程，一门课程有多个学生。使用引用数组避免数据冗余和不一致。

// 学生文档
{ "name": "王五", "courseIds": [ObjectId("..."), ObjectId("...")] }

// 课程文档
{ "name": "数据库原理", "studentIds": [ObjectId("..."), ObjectId("...")] }

4. 嵌入 vs 引用对比表

特性 嵌入 (Embedding) 引用 (Referencing)
数据获取 单次查询获取所有相关数据 多次查询或使用 $lookup
更新 需更新所有包含该数据的副本 只需更新一处
文档大小 可能超过 16MB 限制 无此限制
适用场景 高频访问、少量数据、读取远多于写入 低频访问、大量数据、频繁更新
一致性 可能出现数据副本间不一致 始终一致（单一数据源）
查询复杂度 简单，单文档操作 需要 $lookup 或应用层 join
写入性能 写入较大文档，可能触发碎片整理 写入小文档，性能更优

5. 模式设计模式

以下模式来自 MongoDB 官方文档，用于解决常见的数据建模挑战：

子集模式

Subset Pattern

将频繁访问的数据子集嵌入文档，将其余数据以引用方式存储。适用于"多"的一端数据量较大的场景。

// 电影评论：只嵌入最近5条
{
  "title": "星际穿越",
  "recentReviews": [
    { "user": "A", "score": 5, ... },
    // 最多5条
  ],
  "totalReviewCount": 1250
}
// 完整评论单独存储在 reviews 集合

扩展引用模式

Extended Reference Pattern

将引用文档中高频访问的字段嵌入到当前文档中，避免频繁 $lookup。减少 join 操作。

// 订单文档嵌入客户姓名
{
  "orderId": "1001",
  "customerId": ObjectId("..."),
  "customerName": "张三",  // 冗余但避免join
  "customerLevel": "VIP",  // 高频访问字段
  "items": [...]
}

属性模式

Attribute Pattern

使用键值对数组代替固定字段，解决动态属性问题。可对 attrs.k 和 attrs.v 建立复合索引。

// 商品动态属性
{
  "name": "手机",
  "attrs": [
    { "k": "color", "v": "black" },
    { "k": "storage", "v": "128GB" },
    { "k": "screen", "v": "6.1in" }
  ]
}
// 索引: { "attrs.k": 1, "attrs.v": 1 }

桶模式

Bucket Pattern

将时间序列数据按时间段（小时/天）分组到一个文档中，减少文档数量，提升查询效率。

// 传感器数据按小时分组
{
  "sensorId": "S001",
  "date": ISODate("2024-01-01"),
  "hour": 10,
  "readings": [
    { "t": 600, "v": 23.5 },
    { "t": 660, "v": 24.1 },
    { "t": 720, "v": 23.8 }
  ],
  "minVal": 23.5,
  "maxVal": 24.1
}

多态模式

Polymorphic Pattern

将不同类型的对象存储在同一集合中，通过公共字段统一查询。适合有共同属性但各自有独特字段的场景。

// 产品集合包含不同类型
{ "type": "book", "title": "...", "author": "..." }
{ "type": "laptop", "title": "...", "cpu": "...", "ram": "16GB" }
// 通过 type 字段区分，公共字段统一索引

计算模式

Computed Pattern

预先计算并存储计算结果，避免每次查询时重复计算。适用于读多写少且计算开销大的场景。

// 预计算销售总额
{
  "productId": "P001",
  "name": "商品A",
  "totalSold": 1580,       // 预计算
  "totalRevenue": 79000,  // 预计算
  "monthlyUpdate": ISODate("...")
}

6. 反模式

以下模式在实际项目中应避免使用：

无限增长数组

数组大小没有上限，随时间不断增长。例如将所有日志嵌入用户文档，最终会超过 16MB 文档限制。

使用子集模式限制嵌入数量，或使用引用 + 单独集合

不必要的索引

每个索引都会增加写入开销和存储空间。索引过多会严重拖慢写入性能，且占用大量内存。

只为实际查询创建索引，定期审查并删除未使用的索引

超大文档

文档接近 16MB 限制时，写入性能下降，内存压力增大，WiredTiger 需频繁压缩和解压。

拆分为多个文档，使用引用关联或子集模式

过度范式化

照搬关系型数据库思维，所有关系都用引用表示，导致大量 $lookup 操作，失去文档模型优势。

根据访问模式适当嵌入数据，减少 join 需求
注意：16MB 是 MongoDB 文档的硬性大小限制。设计数据模型时务必评估嵌入数据的增长趋势，避免触碰此限制。

7. 模式验证 ($jsonSchema)

MongoDB 支持使用 $jsonSchema 对集合施加文档验证规则。在数据模型稳定后，这是保障数据质量的关键手段。

db.createCollection("users", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["name", "email"],
      properties: {
        name: {
          bsonType: "string",
          description: "必填，用户姓名"
        },
        email: {
          bsonType: "string",
          pattern: "^.+@.+$",
          description: "必填，合法邮箱格式"
        },
        age: {
          bsonType: "int",
          minimum: 0,
          maximum: 150,
          description: "可选，0~150之间的整数"
        }
      }
    }
  },
  validationLevel: "strict",    // 对所有插入和更新验证
  validationAction: "error"     // 验证失败拒绝操作
})

验证级别与行为

选项 值 说明
validationLevel "strict" 对所有插入和更新操作验证（默认）
"moderate" 仅对已符合验证规则的文档验证更新
"off" 关闭验证
validationAction "error" 验证失败时拒绝操作（默认）
"warn" 验证失败时仅记录警告，允许操作

常用验证操作符

操作符 说明 示例
bsonType 指定字段 BSON 类型 bsonType: "string"
required 必填字段列表 required: ["name", "email"]
pattern 正则表达式匹配 pattern: "^.+@.+$"
minimum / maximum 数值范围约束 minimum: 0, maximum: 150
enum 枚举允许的值 enum: ["active", "inactive"]
additionalProperties 是否允许额外字段 additionalProperties: false
items 数组元素验证 items: { bsonType: "string" }
建议：在开发初期使用 validationAction: "warn" 收集不符合规则的数据情况，待确认规则后再切换为 "error" 严格模式。
