MongoDB 复制

副本集、自动故障转移与高可用性

1. 复制概述

复制（Replication）是 MongoDB 实现高可用性的核心机制，通过在多个服务器上维护数据的副本来提供冗余和故障恢复能力。MongoDB 使用副本集（Replica Set）来实现复制。

复制的核心价值

📈

数据可用性

主节点故障时自动选举新主节点，保证服务持续可用，无需人工干预

📚

读扩展

将读操作分散到从节点，减轻主节点压力，提升整体吞吐量

🌎

数据局部性

将数据部署在靠近用户的位置，降低访问延迟，提升用户体验

🎯

专用目的节点

配置隐藏节点用于报表分析、延迟节点用于灾难恢复，各司其职

提示：生产环境中建议至少部署 3 个数据承载节点（或 2 个数据节点 + 1 个仲裁节点），以确保副本集在任何单节点故障时仍能保持多数派可用。

2. 副本集架构

副本集是一组维护相同数据集的 mongod 进程集合。其中包含一个主节点接收写操作，多个从节点复制主节点的操作日志（oplog）来同步数据。

副本集成员间通信

  • 心跳检测：成员之间每 2 秒互相发送心跳包，用于检测节点存活状态
  • oplog 复制：从节点从主节点的 local.oplog.rs 集合拉取操作日志并重放
  • 选举协议：基于 Raft 共识算法变体，确保最多一个主节点存在
oplog 说明：操作日志（oplog）是一个特殊的固定大小集合（capped collection），记录所有修改数据库数据的操作。所有从节点通过拉取和重放 oplog 来保持与主节点的数据一致。

3. 主节点 (Primary)

主节点是副本集中唯一可以接受写操作的成员。所有的写操作必须指向主节点，主节点将这些操作记录到 oplog 中，从节点通过复制 oplog 来同步数据。

Primary (主节点)
接受所有写操作 • 生成 oplog • 确认 w:"majority" 写入
⬇ oplog 分发
Secondary 1
Secondary 2

主节点的关键特性

  • 唯一写入点：副本集中同一时刻只能有一个主节点接受写操作
  • oplog 生成：主节点将所有写操作记录到 local.oplog.rs 中
  • 多数派确认：只有主节点可以确认 w:"majority" 写入关注级别
  • 选举产生：主节点通过选举协议产生，非固定指定
重要：如果一个副本集的主节点不可用，副本集将通过自动选举产生新的主节点。在选举完成之前，副本集无法接受写操作。

4. 从节点 (Secondary)

从节点维护主节点数据的副本。从节点通过异步方式从主节点的 oplog 中复制操作并在本地重放，从而保持数据同步。

从节点的核心能力

🔄

异步复制

从主节点的 oplog 异步拉取操作日志，在本地重放以同步数据

🎳

参与选举

可被选举为新的主节点（取决于 priority 配置），实现自动故障转移

📖

读操作

通过配置读偏好（Read Preference），客户端可从从节点读取数据

复制流程详解

1. 主节点写入
客户端向主节点发起写操作
⬇
2. 记录 oplog
主节点将操作记录到 local.oplog.rs
⬇
3. 拉取 oplog
从节点长轮询主节点的 oplog 获取新操作
⬇
4. 重放操作
从节点在本地重放 oplog 中的操作，应用数据变更
⬇
5. 确认完成
从节点向主节点确认已应用操作（用于 majority 写关注）
注意：从节点的复制是异步的，这意味着从节点的数据可能略微滞后于主节点。如果读操作需要强一致性，应使用默认的读偏好 primary。

5. 仲裁节点 (Arbiter)

仲裁节点是副本集中的特殊成员，它不存储数据，仅参与选举投票。仲裁节点的存在是为了在偶数个数据节点的情况下打破投票僵局，确保副本集能够顺利完成选举。

仲裁节点的特性

特性 仲裁节点 数据节点
存储数据 否 是
参与选举投票 是 是
可成为主节点 否 是 (priority > 0)

重要限制：仲裁节点是一个永久角色——它永远不会成为主节点或从节点。一个副本集最多只能有一个仲裁节点。不要将仲裁节点部署在与主节点相同的机器上，否则该机器故障会导致两票丢失，副本集可能失去多数派。

6. 自动故障转移

当主节点不可用时，副本集会自动触发选举，从剩余的从节点中选出一个新的主节点，整个过程无需人工干预。

故障转移时间线

主节点故障发生
主节点不可用 (t=0s)
⬇ 等待 electionTimeoutMillis
选举过程 (~10s)
⬇ 新主节点就绪
新主节点上线 (~12s)

关键参数

electionTimeoutMillis 心跳超时时间，默认 10000ms（10秒）。从节点在超过此时间未收到主节点心跳后触发选举。
平均选举时间 约 12 秒（包含心跳超时 + 选举协商）。MongoDB 5.0+ 引入镜像读取以加速选举。

可重试写入

MongoDB 驱动支持可重试写入（Retryable Writes），当写操作因主节点故障而失败时，驱动会自动在新主节点选举完成后重试该操作，对应用透明。

🔁

镜像读取 (Mirror Reads)

MongoDB 5.0+ 在选举前将部分读取请求镜像发送给从节点，帮助从节点提前预热缓存，加速选举后的读取性能

🔄

可重试写入

驱动自动在故障转移后重试写操作，应用无需额外处理选举期间的短暂不可用

最佳实践：设置 electionTimeoutMillis 时需权衡：值太小可能导致网络抖动引起不必要的选举；值太大则故障恢复时间过长。建议保持默认值或根据网络环境适当调整。

7. 异步复制详解

MongoDB 的复制是异步的——从节点以异步方式从主节点的 oplog 中拉取操作日志并在本地重放。这种设计在保证可用性的同时允许一定的复制延迟。

复制延迟监控

复制延迟（Replication Lag）是从节点数据落后于主节点的程度，是副本集健康度的关键指标。

延迟仪表盘示意

正常 (<1s)
警告 (1-10s)
严重 (>10s)

流控制机制

流控制（Flow Control）是 MongoDB 4.2+ 引入的机制，当复制延迟超过阈值时自动限制主节点的写入速率，防止延迟进一步扩大。

flowControlTargetLagSeconds 目标延迟阈值，默认 10 秒。当实际延迟超过此值时，主节点开始限制写入速率。
flowControlThresholdMs 流控制采样间隔，默认 1000ms。MongoDB 定期采样延迟并决定是否启用流控制。
注意：如果从节点的延迟持续增长且流控制无法有效抑制，应检查：网络带宽、磁盘 I/O、从节点硬件配置以及是否有长时间运行的查询阻塞了 oplog 应用。

监控命令

// 查看副本集状态，包含各从节点延迟
rs.status()

// 查看复制延迟（秒）
rs.printSecondaryReplicationInfo()

// 查看当前 oplog 窗口大小
use local
db.oplog.rs.stats()

8. 读偏好 (Read Preference)

读偏好决定了客户端将读操作路由到副本集的哪个成员。默认情况下，所有读操作都指向主节点以保证强一致性，但可以通过配置读偏好来实现读扩展和降低延迟。

模式 说明 适用场景
primary 只从主节点读取 默认模式，强一致性需求，对数据时效性要求严格
primaryPreferred 优先从主节点读取，主不可用时回退到从节点 需要一致性但可容忍主节点故障时的降级读取
secondary 只从从节点读取 报表/分析场景，分担主节点负载，可接受最终一致性
secondaryPreferred 优先从从节点读取，无从节点可用时回退到主节点 读扩展场景，主节点作为最后回退选项
nearest 从网络延迟最低的节点读取（无论主从） 低延迟读取需求，多数据中心部署场景

读偏好选择决策流程

需要强一致性吗？
⬇ 是
primary 或 primaryPreferred
需要读扩展吗？
⬇ 是
secondary 或 secondaryPreferred
需要最低延迟吗？

提示：读偏好可与 maxStalenessSeconds 配合使用，限制从节点的最大延迟。还可与 tag sets 配合，将读操作路由到特定标签的节点（如按数据中心、硬件类型等分类）。

9. 特殊从节点类型

从节点可以通过配置不同的属性，承担不同的专用角色，满足特定业务需求。

类型 priority hidden 说明
优先级0节点
(Priority 0)		 0 no 永远不会成为主节点，但数据完整可用，适合部署在异地数据中心或用于读扩展
隐藏节点
(Hidden)		 0 yes 对客户端不可见（不会出现在 isMaster() 结果中），用于专用任务如报表、备份
延迟节点
(Delayed)		 0 yes 延迟应用 oplog（如 3600 秒），提供历史数据快照，用于灾难恢复和人为误操作回退

配置示例

// 优先级0节点（异地数据中心，不参与选举）
var cfg = rs.conf();
cfg.members[2].priority = 0;
rs.reconfig(cfg);

// 隐藏节点（专用报表，对客户端不可见）
var cfg = rs.conf();
cfg.members[3].priority = 0;
cfg.members[3].hidden = true;
rs.reconfig(cfg);

// 延迟节点（延迟1小时应用oplog，灾难恢复）
var cfg = rs.conf();
cfg.members[4].priority = 0;
cfg.members[4].hidden = true;
cfg.members[4].slaveDelay = 3600;  // 秒
rs.reconfig(cfg);
🔒

优先级0节点

数据完整，可读，不参与选举成为主节点，适合异地灾备

🕵

隐藏节点

客户端感知不到，不影响应用连接逻辑，专用于后台分析任务

⏲

延迟节点

数据保留在历史时间点，人为误删数据时可从延迟节点恢复

警告：延迟节点必须是隐藏节点且优先级为0，因为它的数据是延迟的，不能被选举为主节点，也不应被客户端直接读取（除非显式配置读偏好）。

10. 数据可见性与读关注

读关注（Read Concern）级别控制从副本集读取数据时的一致性和隔离级别。不同级别在一致性和性能之间做出不同权衡。

"local"
返回实例最新数据，可能包含未被多数派确认的写入。默认读关注级别。
"available"
返回实例最新数据（与 local 相似），但不保证因果一致性。适用于分片集群读取 orphans 数据的场景。
"majority"
返回已被副本集多数派确认的数据，保证不会被回滚。最强一致性保证。

读关注与读偏好的交互

读偏好 "local" "available" "majority"
primary 主节点最新数据 主节点最新数据 多数派确认数据
secondary 从节点最新数据（可能被回滚） 从节点最新数据（无因果保证） 多数派确认数据
nearest 最近节点最新数据 最近节点最新数据 多数派确认数据

副本集中的事务

MongoDB 4.0+ 支持副本集上的多文档事务（分片集群事务从 4.2 开始支持）。事务提供了更强的数据一致性保证：

  • 事务内的操作具有原子性：要么全部提交，要么全部回滚
  • 事务默认使用 "majority" 读关注，确保读取多数派确认的数据
  • 事务支持快照隔离，事务开始时看到的数据一致性快照在整个事务期间保持不变
  • 事务的写关注可以配置为 w:"majority"，确保事务提交在多数派节点持久化
// 副本集事务示例
const session = db.getMongo().startSession();
session.startTransaction({
    readConcern: { level: "majority" },
    writeConcern: { w: "majority" }
});

try {
    const accounts = session.getDatabase("bank").accounts;
    accounts.updateOne({ _id: 1 }, { $inc: { balance: -100 } });
    accounts.updateOne({ _id: 2 }, { $inc: { balance: 100 } });
    session.commitTransaction();
} catch (e) {
    session.abortTransaction();
    throw e;
} finally {
    session.endSession();
}
注意："local" 和 "available" 读关注可能返回尚未被多数派确认的数据。如果主节点在写操作被多数派确认前崩溃，这些数据可能被回滚。对数据一致性有严格要求的场景应使用 "majority"。

11. 多数据中心部署

将副本集成员分布到多个数据中心，可以提高容灾能力和数据局部性。MongoDB 支持跨数据中心的副本集部署，包括同地域多可用区和跨地域部署。

部署策略

🏗

对称部署

每个数据中心部署相同数量的节点，任一数据中心故障不影响多数派

🌍

主从部署

主数据中心部署多数节点，从数据中心部署少数节点+仲裁节点

🔐

优先级控制

通过设置不同数据中心的节点优先级，控制主节点的选举倾向

多数据中心部署要点

  • 多数派保证：确保任一数据中心故障后，剩余节点仍能构成多数派（大于总票数的一半）
  • 仲裁节点放置：将仲裁节点部署在第三个数据中心，用于打破平局
  • 网络延迟：跨数据中心复制会增加延迟，影响写性能和选举速度
  • 读偏好标签：使用 tag sets 将读操作路由到最近的节点，降低访问延迟
关键原则：确保多数派节点不在同一个数据中心内。如果多数派节点集中在同一数据中心，该数据中心故障将导致整个副本集失去写能力。建议采用 3+2+1 或 2+2+1 的分布模式。 
// 配置跨数据中心节点的优先级
var cfg = rs.conf();
// 数据中心A的节点优先级高，优先成为主节点
cfg.members[0].priority = 2;  // DC-A
cfg.members[1].priority = 2;  // DC-A
// 数据中心B的节点优先级低，通常不成为主节点
cfg.members[2].priority = 1;  // DC-B
cfg.members[3].priority = 1;  // DC-B
// 数据中心C仅仲裁
cfg.members[4].arbiterOnly = true;  // DC-C
rs.reconfig(cfg);
