MongoDB 分片

Sharding - 水平扩展与数据分布

1. 分片概述

分片（Sharding）是 MongoDB 将数据水平分布到多台机器上的方法。当单台服务器的资源（CPU、内存、磁盘I/O）无法满足吞吐量或数据量需求时，分片允许通过增加机器来扩展数据库容量。

垂直扩展 vs 水平扩展

方式 说明 优缺点
垂直扩展 增加单机配置（CPU、内存、SSD等） 简单直接，但存在硬件上限，成本递增
水平扩展 增加机器数量，将数据分散存储 无理论上限，但系统架构更复杂
何时使用分片？ 当数据集或吞吐量超过单机容量，且复制集无法提供足够的写入扩展能力时，应考虑分片。分片不应用于早期的"小规模"部署。

3. 核心组件

Shard（分片）

每个分片包含集群数据的一个子集。每个分片必须是一个副本集，以提供分片级别的高可用性和数据冗余。在生产环境中，所有分片都应为副本集。

mongos（查询路由器）

mongos 是客户端应用程序与分片集群之间的接口。mongos 将查询请求路由到正确的分片，然后将结果合并返回给客户端。应用程序绝不应直接连接分片，而应始终通过 mongos 连接。

  • 解析客户端查询，确定目标分片
  • 从配置服务器获取集群元数据（分片键范围映射）
  • 将查询路由到相关分片并合并结果
  • 支持部署多个 mongos 实例以实现高可用

Config Servers（配置服务器）

配置服务器存储分片集群的元数据和配置信息，包括：

  • 每个分片上的数据段（chunk）范围
  • 集群中可用的分片列表
  • 分片键与数据库/集合的映射关系
重要：配置服务器必须是副本集（CSRS - Config Server Replica Set）。配置服务器的数据对集群运行至关重要，如果配置服务器数据丢失，集群将无法正常工作。

4. 分片键（Shard Key）

分片键决定了文档在集群中的分布方式。MongoDB 使用分片键的值将文档划分到不同的数据段（chunk）中，进而分配到不同的分片上。

分片键的类型

  • 单字段分片键：基于一个字段，如 userId
  • 复合分片键：基于多个字段，如 { userId: 1, createdAt: 1 }

分片键的限制

  • 分片键在集合创建后不可更改（直到 MongoDB 5.0 引入重新分片功能）
  • 分片键字段的值不可变：已分片集合中不可修改分片键字段的值
  • 缺失的分片键值被视为 null，用于数据分布计算
  • 分片键索引必须存在且不能是唯一索引（除非是 _id 或范围分片的单个键）

重新分片（Resharding）- MongoDB 5.0+

MongoDB 5.0 引入了重新分片功能，允许更改集合的分片键。重新分片过程在后台进行，集群在此期间仍可正常读写。

分片键精化（Refinement）- MongoDB 4.4+

从 4.4 开始，可以通过在现有分片键上添加后缀字段来精化分片键。这在原有分片键的基数不足时非常有用，且不需要完全重新分片。

关于缺失的分片键：如果文档不包含分片键字段，该字段在数据分布时被视为 null。这意味着大量缺失分片键的文档会被集中到同一数据段中，导致数据分布不均。

分片键选择指南

高基数
分片键应有足够多的不同值，以确保数据可以均匀分布。例如：userId、orderId
低频率
每个分片键值不应出现过高的频率，避免单个值产生过大的数据段
非单调递增
避免使用如 ObjectId 或时间戳等单调递增的值作为范围分片键，会导致所有写入集中到最后一个分片
低基数
如布尔值（只有 true/false），最多只能分布到有限的数据段中
高频率值
某个值出现过多会导致"超大块"（jumbo chunk），无法迁移
单调递增
如用时间戳做范围分片键，所有新数据写入同一分片，热点问题严重

5. 数据段（Chunks）

MongoDB 将分片集合的数据划分为若干数据段（chunk）。每个数据段对应分片键值范围中的一个子区间。

数据段的范围规则

每个数据段定义了分片键值的范围：下界包含，上界不包含（左闭右开区间）。

数据段 A
[-inf, 100)
在 Shard 1 上
→
数据段 B
[100, 500)
在 Shard 2 上
→
数据段 C
[500, +inf)
在 Shard 3 上

数据段拆分

当数据段的大小超过配置的阈值（默认 64MB）时，MongoDB 会自动将数据段拆分为两个较小的数据段。拆分基于分片键值的中点进行。

数据段迁移

均衡器（Balancer）会在分片间迁移数据段以实现均匀分布。迁移过程在后台进行，对应用程序透明。

超大块（Jumbo Chunk）：当一个数据段的大小超过阈值但无法拆分（因为所有文档的分片键值都相同）时，该数据段被标记为"超大块"。超大块无法被均衡器迁移，需要手动处理。

6. 均衡器（Balancer）

均衡器是一个后台进程，负责管理数据段在分片间的迁移，确保数据均匀分布。

均衡器的工作机制

  • 均衡器运行在 mongos 实例上（主 mongos）
  • 定期检查各分片上的数据段数量
  • 当数据段分布不均匀时，自动发起迁移
  • 迁移过程中源分片和目标分片继续正常提供服务

自动均衡的触发

  • shardCollection 命令会启动均衡器
  • 添加新分片时，均衡器会自动将数据迁移到新分片
  • 数据段拆分后，可能触发后续迁移

即时均衡 - shardAndDistributeCollection (8.0+)

MongoDB 8.0 引入了 shardAndDistributeCollection 命令，在分片集合的同时立即将数据均匀分布到所有分片上，而非等待后台均衡器逐步完成。对于大型集合，这可以显著缩短数据均衡的时间。

shardCollection
创建初始数据段，启动均衡器后台迁移
vs
shardAndDistributeCollection 8.0+
分片同时立即均衡数据分布

7. 分片策略

策略 说明 优点 缺点
哈希分片
hashed		 对分片键值进行哈希计算，按哈希值分配数据段 数据分布更加均匀，写入负载均衡 范围查询需广播到所有分片
范围分片
1 / -1		 按分片键值的范围分配数据段 支持针对分片键的定向范围查询 数据可能分布不均匀（热点）

哈希分片详细说明

哈希分片使用分片键字段的哈希值来分配文档。由于哈希函数将接近的值映射到随机位置，数据分布更加均匀，但牺牲了范围查询的效率。

范围分片详细说明

范围分片按分片键值的数值或字典顺序将数据划分为连续区间。值的"相近"意味着在物理上更可能位于同一分片上，这有利于范围查询的定向路由，但如果分片键单调递增则会导致写入热点。

8. 定向操作与广播操作

分片集群中，查询操作的效率取决于查询是否包含分片键：

定向操作（Targeted Operation）

当查询条件包含分片键时，mongos 可以确定目标分片，将请求直接路由到对应的分片。这是最高效的查询方式。

定向查询
客户端
→
mongos
→
Shard 2
查询条件包含分片键 → mongos 精准路由到目标分片

广播操作 / 散集收集（Broadcast / Scatter-Gather）

当查询条件不包含分片键时，mongos 无法确定目标分片，必须将查询广播到所有分片，然后收集并合并结果。这会显著增加延迟和资源消耗。

广播查询
客户端
→
mongos
→
Shard 1
Shard 2
Shard 3
查询条件不含分片键 → mongos 广播到所有分片，收集合并结果
性能提示：尽可能让查询条件包含分片键，以利用定向路由避免广播查询的开销。对于无法包含分片键的高频查询，应考虑建立合适的复合索引或重新评估分片键选择。

9. 区域（Zones）

区域（Zones）允许将分片键的特定值范围与特定的分片关联起来，实现数据局部性管理。这在地理分布式部署中特别有用。

区域工作原理

  1. 为分片定义区域（如 "Asia"、"Europe"）
  2. 为集合定义区域范围（zone range），指定分片键值范围与区域的映射
  3. 均衡器自动将匹配范围的数据段迁移到关联的分片上
分片键范围 → 区域 → 分片
Zone: Asia
region: ["CN", "JP"]
↓ 路由到
Shard-Asia
Zone: Europe
region: ["DE", "UK"]
↓ 路由到
Shard-EU
Zone: Americas
region: ["US", "BR"]
↓ 路由到
Shard-US
区域与均衡器：区域不替代均衡器，均衡器仍然会确保分片间的数据段数量大致均衡，但会优先遵守区域约束。未被区域覆盖的数据段可以分配到任意分片。

10. 分片与未分片集合

分片集群中的数据库可以同时包含分片集合和未分片集合：

未分片集合

未分片集合的所有数据存储在该数据库的主分片（Primary Shard）上。主分片是在启用分片时自动选择的，与复制集中的 primary 节点无直接关系。

分片集合

分片集合的数据分布在集群的多个分片上，由分片键决定分布方式。

特性 未分片集合 分片集合
数据存储位置 数据库主分片 分布在多个分片上
查询路由 路由到主分片 根据分片键路由或广播
写入扩展 × ✓
均衡器管理 × ✓
重要规则：无论操作分片还是未分片集合，应用程序都必须通过 mongos 连接集群。绝不应直接连接分片节点进行操作，否则可能导致数据不一致或查询结果不完整。

11. 连接规则

必须通过 mongos 连接

所有客户端应用程序都必须通过 mongos 实例连接到分片集群。mongos 接受客户端请求并路由到正确的分片，然后将结果合并后返回。

连接字符串示例

mongodb://mongos1.example.com:27017,mongos2.example.com:27017,mongos3.example.com:27017/?replicaSet=myCluster

禁止直连分片

  • 直接连接分片可能导致查询结果不完整（只看到一个分片的数据）
  • 直连写入可能破坏集群元数据一致性
  • 直连无法利用跨分片事务和路由功能
运维注意：在运维场景下（如备份单个分片），可以直连分片的 primary 节点，但应仅限于受控的维护操作，而非应用正常的读写请求。

12. DDL 操作限制

在分片集群中，部分 DDL（数据定义语言）操作受到限制或需要特殊处理：

操作 限制说明 版本
dropDatabase 必须在 mongos 上执行，会删除所有分片上的数据库数据 全版本
dropCollection 必须在 mongos 上执行，对分片集合会清理所有分片上的数据和元数据 全版本
createCollection 分片集合的创建需使用 shardCollection 命令 全版本
唯一索引 分片集合上的唯一索引必须包含分片键作为前缀（例外：_id 索引） 全版本
跨分片事务 支持跨分片事务，但性能开销较大，应尽量避免 4.2+
reshardCollection 允许更改集合的分片键，操作在后台进行 5.0+
shardAndDistributeCollection 分片的同时立即均衡分布数据 8.0+
细化分片键 可添加后缀字段到现有分片键（精化），无需完全重新分片 4.4+
moveCollection 将未分片集合从一个分片移动到另一个分片 8.3+
_id 索引的特殊规则：分片集合的 _id 索引必须是唯一索引，但不要求包含分片键。MongoDB 默认在 _id 上创建唯一索引。如果应用程序自行指定 _id 值，则需确保跨分片的唯一性。
