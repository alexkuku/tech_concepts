时间序列集合

Time Series Collections - 高效存储与查询时序数据

概述

时间序列集合（Time Series Collections）是 MongoDB 5.0 引入的专门用于存储和查询时间序列数据的集合类型。针对时序数据的特征进行了底层存储优化，在写入性能、查询效率和存储空间方面相比普通集合具有显著优势。

版本要求：时间序列集合从 MongoDB 5.0 开始可用。自动复合索引功能需要 6.3 及以上版本。

数据组成

时间序列数据由三个核心组成部分：

时间字段
timeField
数据点的时间戳
元数据字段
metaField
数据源的标识信息
测量值
Measurements
实际采集的数值
组成部分 说明 示例
时间字段 (timeField) 每个数据点必须包含一个时间戳，指定为 Date 类型 timestamp: ISODate("2024-01-15T10:30:00Z")
元数据字段 (metaField) 可选。标识数据源的元信息，用于分组和过滤 metadata: { sensorId: "S001", region: "east" }
测量值 实际采集的数值数据，存储在文档的其余字段中 temperature: 23.5, humidity: 65

创建时间序列集合

// 基本创建
db.createCollection("weather", {
  timeseries: {
    timeField: "timestamp",
    metaField: "metadata",
    granularity: "hours"
  }
})

// 完整选项
db.createCollection("sensorData", {
  timeseries: {
    timeField: "timestamp",       // 必填：时间字段名
    metaField: "sensorMetadata",  // 可选：元数据字段名
    granularity: "minutes",       // 可选：时间粒度 (seconds/minutes/hours)
    bucketMaxSpanSeconds: 3600,   // 可选：桶最大时间跨度
    bucketRoundingSeconds: 3600   // 可选：桶起始时间取整
  },
  expireAfterSeconds: 86400       // 可选：自动过期时间（TTL）
})

granularity 选项说明

粒度值 含义 适用场景 默认桶跨度
"seconds" 数据点间隔秒级 高频采集、实时监控 3600 秒（1 小时）
"minutes" 数据点间隔分钟级 常规传感器、业务指标 86400 秒（1 天）
"hours" 数据点间隔小时级 每日统计、趋势分析 2592000 秒（30 天）
重要：granularity 应与实际数据采集频率匹配。设置不当会影响压缩和查询性能。如果数据入库频率与采集频率不同步，可使用 bucketMaxSpanSeconds 和 bucketRoundingSeconds 进行精细调整。

内部存储格式

时间序列集合在内部使用列式存储格式（columnar storage format）优化数据组织。MongoDB 将同一元数据分组和时间范围内的数据点压缩到桶（bucket）结构中，以减少存储开销并提升查询性能。

写入的时序文档
{ timestamp, metadata, value1, value2 }
↓ 自动桶压缩
内部桶文档 (Bucket)
meta: { sensorId: "S001" }
data: {
  timestamp: [t1, t2, t3, ...],
  temperature: [23.5, 24.1, 23.8, ...],
  humidity: [65, 63, 64, ...]
}
control: { count: 3, min/max: ... }
↓ 列式存储
WiredTiger 列式优化
同类型数据连续存储，提升压缩率和扫描效率
优势：列式存储使得同类数据（如所有温度值）连续存储在磁盘上，大幅提升压缩率（通常 2-10x）和范围查询效率。

更新限制

时间序列集合对更新操作有严格限制，以保护内部桶结构的完整性：

操作 限制 原因
更新测量值字段 不允许 测量值存储在桶内部，修改可能破坏桶结构
更新 metaField 允许 metaField 是桶的分组键，可以修改但会导致文档重新分配到新桶
更新 timeField 不允许 时间字段决定桶的分配，修改会破坏时间有序性
删除文档 仅能按 metaField 删除（6.0+） 部分删除需匹配元数据条件 
// 允许：更新 metaField
db.weather.updateOne(
  { "metadata.sensorId": "S001" },
  { $set: { "metadata.region": "north" } }
)

// 不允许：更新测量值
db.weather.updateOne(
  { timestamp: ISODate("2024-01-15T10:30:00Z") },
  { $set: { temperature: 25.0 } }  // 报错!
)

与普通集合对比

时间序列集合相比普通集合在时序数据场景下具有显著优势：

对比维度 普通集合 时间序列集合
数据处理复杂度 需要手动实现桶压缩和聚合逻辑 自动桶压缩，开箱即用
查询效率 需要手动优化索引和聚合管道 内置列式存储优化，时序查询更快
磁盘空间 每个文档独立存储元数据，冗余度高 桶内共享元数据，压缩率 2-10x
磁盘 I/O 查询需扫描大量独立文档 列式存储减少 IO，仅读取需要的列
缓存利用率 缓存中包含大量重复元数据 共享元数据节省缓存空间
写入性能 逐文档写入 批量写入桶，减少写入放大

索引策略

时间序列集合的索引策略与普通集合有所不同：

自动复合索引

从 MongoDB 6.3 开始，创建时间序列集合时自动生成 metaField + timeField 的复合索引。

// 自动创建的索引（6.3+）
// 等价于：
db.sensorData.createIndex({ "sensorMetadata": 1, "timestamp": 1 })

手动添加二级索引

// 在 metaField 子字段上添加索引
db.sensorData.createIndex({
  "sensorMetadata.region": 1,
  "timestamp": 1
})

// 在测量值字段上添加索引（用于排序和范围查询）
db.sensorData.createIndex({
  "temperature": 1,
  "timestamp": 1
})
建议：对于时序查询，metaField + timeField 复合索引通常是最有效的查询模式。如果经常按 metaField 子字段过滤，可添加对应二级索引。

分片支持

分片策略 是否支持 说明
基于 metaField 哈希分片 支持 推荐方式，按元数据均匀分布数据
基于 metaField 范围分片 支持 适用于元数据有自然分区的场景
基于 timeField 分片 不支持 时间单调递增会导致热块问题
Zone 分片 不支持 时间序列集合不支持 Zone/Tag 感知分片 
// 对时间序列集合启用分片
sh.enableSharding("mydb")
sh.shardCollection("mydb.sensorData", { "sensorMetadata": 1 })
注意：Zone 分片（Zone Sharding）当前不支持时间序列集合。如果需要地理分布数据，建议在应用层实现路由逻辑。

应用场景

行业领域 场景描述 典型数据 metaField 示例
物联网 (IoT) 传感器数据采集与监控 温度、湿度、压力、位置 { sensorId, deviceId, location }
金融 股票行情、交易流水 价格、成交量、汇率 { ticker, exchange, symbol }
零售 销售数据、库存变动 销售额、库存量、订单数 { storeId, productId, category }
DevOps 系统监控、应用性能 CPU、内存、QPS、延迟 { host, service, region }

不适用场景

时间序列集合并非所有场景的最佳选择，以下情况应使用普通集合：

不适用场景 原因 推荐替代方案
无序数据 时间序列集合假设数据按时间有序入库，无序数据无法有效桶压缩 普通集合 + 索引
非时间依赖数据 没有时间维度的数据不适合时序模型 普通集合 / 问答文档
频繁更新测量值 时序集合不允许更新测量值字段 普通集合
需要 Zone 分片 时序集合不支持 Zone 分片 普通集合 + Zone Sharding

文档结构示意

以下是时间序列集合中文档的完整结构示例：

时序文档结构
{
  timestamp: ISODate("2024-01-15T10:30:00Z"),  // timeField
  metadata: {                        // metaField
    sensorId: "S001",
    region: "east"
  },
  temperature: 23.5,                      // 测量值
  humidity: 65                           // 测量值
}
↓
内部桶文档结构 (Bucket)
{
  _id: ObjectId("..."),
  meta: { sensorId: "S001", region: "east" },
  data: {
    timestamp: { 0: t1, 1: t2, 2: t3 },
    temperature: { 0: 23.5, 1: 24.1, 2: 23.8 },
    humidity: { 0: 65, 1: 63, 2: 64 }
  },
  control: {
    count: 3,
    min: { timestamp: t1, temperature: 23.5 },
    max: { timestamp: t3, temperature: 24.1 }
  }
}
关键点：对外部查询而言，时序集合与普通集合的使用体验一致。MongoDB 在查询时自动将桶文档拆解还原为原始文档格式，对应用透明。
