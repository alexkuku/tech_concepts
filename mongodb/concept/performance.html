性能优化

Performance - 诊断与优化策略

常见性能问题

MongoDB 性能问题的常见原因可以归纳为以下几类：

问题类别 具体原因 典型表现
不合理的访问策略 全表扫描、缺少索引查询、返回过多字段 查询响应时间长、CPU 飙升
硬件资源不足 内存不够、磁盘 I/O 慢、CPU 不足 系统级延迟、页面故障多
连接数过多 连接池配置不当、连接泄漏 连接排队、响应超时
不合理的索引 冗余索引、缺失索引、索引未覆盖查询 写入变慢、查询走错执行计划
不合理的 Schema 设计 过度嵌入或过度引用、大文档、无限增长数组 文档移动、写入放大

性能排障流程

当遇到性能问题时，按照以下流程逐步诊断：

1. 确认性能问题表现
慢查询? 高CPU? 高内存? 连接超时?
↓
2. 检查服务器状态
serverStatus / dbStats / collStats
↓
3. 分析连接情况
connections.current / connections.available
↓
4. 检查锁与队列
acquireWaitCount / currentQueue
↓
5. 分析慢查询
Profiler / Explain / mongotop / mongostat
↓
6. 优化索引与 Schema
添加/删除索引、调整 Schema
↓
7. 评估硬件资源
内存、磁盘 I/O、CPU
↓
8. 验证优化效果
重新运行负载测试、对比指标

锁分析

MongoDB 使用多粒度锁机制。通过 serverStatus 命令可以获取锁相关的关键指标：

db.serverStatus().locks
db.serverStatus().globalLock

关键锁指标

指标 含义 关注点
acquireWaitCount 锁等待次数 值高表示锁争用严重
timeAcquiringMicros 获取锁的累计等待时间（微秒） 值高表示锁等待时间长
deadlockCount 死锁检测次数 应为 0，非零表示需要排查
currentQueue 当前等待队列长度 包含 reader 和 writer 子字段
activeClients 活跃客户端数 包含 reader 和 writer 子字段
提示：在 WiredTiger 引擎中，默认使用文档级锁，锁争用通常较少。但如果观察到 currentQueue 持续增长，需要重点检查是否有长时间运行的写操作或全表扫描。

锁类型层级

全局锁 (Global)
↓
数据库锁 (Database)
↓
集合锁 (Collection)
↓
文档锁 (Document) - WiredTiger 默认

慢查询分析

慢查询是性能问题最常见的表现形式，主要原因包括：

原因 说明 诊断方法
无效索引 查询未使用索引或使用了错误的索引 explain() 查看 winningPlan
不合理的 Schema 大文档、无限增长数组导致文档移动 collStats 查看平均文档大小
不合理的查询结构 $or / $in 使用不当、正则表达式未锚定 分析查询模式，重写查询
内存不足 工作集大于可用内存，频繁页面故障 serverStatus().metrics 查看 page faults

常用诊断工具

# 实时监控
mongostat --host localhost --port 27017

# 集合级别耗时分析
mongotop --host localhost --port 27017

# 查看当前正在执行的操作
db.currentOp(true)

# 终止长时间运行的操作
db.killOp(opId)
注意：生产环境中应谨慎终止操作，确保不会破坏正在进行的写操作一致性。优先排查慢查询的根本原因。

连接分析

连接数管理对 MongoDB 性能至关重要。通过 serverStatus 监控连接指标：

db.serverStatus().connections
指标 含义 建议
connections.current 当前活跃连接数 不应接近上限
connections.available 剩余可用连接数 应保持充足余量
connections.totalCreated 累计创建的连接总数 增长过快可能有连接泄漏

连接池配置建议

参数 默认值 建议
maxPoolSize 100 根据并发量调整，通常 50-200
minPoolSize 0 设置合理最小值以避免冷启动
maxIdleTimeMS 无 设置超时以回收空闲连接
waitQueueTimeoutMS 无 设置等待超时避免无限等待
公式参考：总连接数上限 = 应用实例数 × maxPoolSize + 系统预留（监控、备份等）。确保不超过 MongoDB 的 ulimit 和 maxIncomingConnections 限制。

执行计划 (Explain)

explain() 是诊断查询性能的核心工具，提供三种详细程度：

// 基本用法
db.collection.find({ status: "active" }).explain()

// 三种详细程度
db.collection.find({ status: "active" }).explain("queryPlanner")
db.collection.find({ status: "active" }).explain("executionStats")
db.collection.find({ status: "active" }).explain("allPlansExecution")
模式 包含信息 用途
"queryPlanner" 查询计划选择过程、winningPlan、rejectedPlans 查看是否走了正确的索引
"executionStats" 包含 queryPlanner + 实际执行统计 查看扫描行数、执行时间等指标
"allPlansExecution" 包含 executionStats + 所有候选计划的执行统计 对比不同计划的执行效果

关键执行指标

指标 含义 理想值
totalDocsExamined 扫描的文档数 等于返回文档数
totalKeysExamined 扫描的索引键数 等于返回文档数
nReturned 返回的文档数 -
executionTimeMillis 执行时间（毫秒） 尽可能低
stage 执行阶段（COLLSCAN / IXSCAN / FETCH 等） 应避免 COLLSCAN
高效查询公式：nReturned ≈ totalKeysExamined ≈ totalDocsExamined。如果 totalDocsExamined 远大于 nReturned，考虑添加索引或使用投影来减少扫描量。

性能分析器 (Profiler)

MongoDB 的数据库分析器（Profiler）可以记录操作的详细信息，用于诊断性能问题。

// 设置分析级别
db.setProfilingLevel(0)                                    // 关闭（默认）
db.setProfilingLevel(1, { slowms: 100 })                  // 仅记录慢操作（>100ms）
db.setProfilingLevel(2)                                    // 记录所有操作

// 查看当前级别
db.getProfilingStatus()

// 查询分析结果
db.system.profile.find().sort({ ts: -1 }).limit(10)

// 查询特定集合的慢操作
db.system.profile.find({
  ns: "mydb.orders",
  millis: { $gt: 50 }
}).sort({ ts: -1 })
级别 值 行为 性能影响
关闭 0 不记录任何操作 无
慢操作 1 记录超过 slowms 阈值的操作 轻微
全部 2 记录所有操作 较大，仅用于调试
警告：级别 2 会记录所有操作，包括分析器自身的写入，可能严重影响性能。仅在排查问题时临时开启，排查完毕后及时关闭。

优化策略

根据诊断结果，采取以下优化策略：

1. 合理创建索引

// ESR 原则：等值(E) → 排序(S) → 范围(R)
db.orders.createIndex({
  userId: 1,        // 等值查询
  createdAt: -1,    // 排序
  amount: 1         // 范围查询
})

// 查看冗余索引
db.orders.getIndexes()

// 删除冗余索引（如 {a:1} 被 {a:1,b:1} 覆盖）
db.orders.dropIndex("a_1")

2. 覆盖查询

// 非覆盖查询：需要回表 FETCH
db.users.find({ name: "Alice" }, { name: 1, email: 1 })
// 索引: { name: 1, email: 1 } 可以覆盖此查询

// 覆盖查询：所有字段都在索引中，无需回表
db.users.find(
  { name: "Alice" },
  { _id: 0, name: 1, email: 1 }  // 排除 _id 才能真正覆盖
)
// explain() 中 stage 为 "PROJECTION" 而非 "FETCH"

3. 投影优化

// 仅返回需要的字段，减少网络传输和内存占用
db.orders.find(
  { status: "active" },
  { orderId: 1, total: 1, _id: 0 }
)

4. 排序使用索引

// 无索引排序（in-memory sort，超过 32MB 报错）
db.orders.find().sort({ createdAt: -1 })

// 有索引排序（索引有序，避免内存排序）
db.orders.createIndex({ createdAt: -1 })
db.orders.find().sort({ createdAt: -1 })

优化策略总结

策略 原理 效果
合理索引 减少扫描行数，避免全表扫描 大幅减少 IO 和 CPU 开销
覆盖查询 仅扫描索引，无需回表 减少 IO，提升查询速度
投影限制 仅返回必要字段 减少网络传输和内存
索引排序 利用索引有序性避免内存排序 避免 32MB 排序内存限制
Schema 优化 合理嵌入/引用、控制文档大小 减少文档移动和写入放大
硬件升级 增加内存、使用 SSD 减少页面故障、提升 IO 速度
