MongoDB 索引

加速查询的核心机制 -- 理解索引结构、类型与策略

1. 索引概述

索引是 MongoDB 中提高查询效率的核心数据结构。类似于书籍的目录，索引让数据库无需扫描整个集合就能快速定位到所需文档，将查询时间从 O(n) 降低到 O(log n)。

B-Tree 数据结构

MongoDB 使用 B-Tree（B树）作为索引的底层数据结构（从 MongoDB 4.2 起统一使用 B-Tree，替代了早期的 Hash 索引方案用于大多数场景）。B-Tree 的核心特性：

  • 平衡多路搜索树 -- 所有叶子节点处于同一层级，查询路径长度一致
  • 有序存储 -- 索引键按排序顺序存储，支持范围查询和排序
  • 高扇出低高度 -- 每个节点可存储大量键值，树高度通常只有 3-4 层

读写权衡

索引是有代价的。每个索引都会占用额外的磁盘和内存空间，并在每次写操作（插入、更新、删除）时需要同步维护索引结构。

读性能
提升 75%
索引越多，查询越快
写性能
降低 35%
索引越多，写入越慢

默认 _id 索引

每个集合在创建时自动在 _id 字段上建立唯一索引，该索引不可删除。_id 索引确保每个文档有唯一标识，是复制集和分片集群中数据同步的关键依据。

要点：MongoDB 的 _id 默认是 ObjectId 类型，12 字节，包含时间戳、机器标识和计数器信息。_id 也可以使用任何唯一值（如 UUID、自增数字等）。

2. 索引类型

MongoDB 提供多种索引类型，以支持不同场景的查询需求。选择正确的索引类型是优化性能的关键。

索引类型 说明 创建示例
单字段索引 在单个字段上建立索引，最基础的索引类型。1 表示升序，-1 表示降序 db.col.createIndex({name: 1})
复合索引 在多个字段上建立的组合索引。字段顺序影响索引效率，需遵循 ESR 规则 db.col.createIndex({name: 1, age: -1})
多键索引 当索引字段为数组类型时自动创建。为数组中的每个元素建立独立索引项 db.col.createIndex({"tags": 1})
文本索引 支持对字符串内容进行全文搜索，支持词干提取和停用词过滤 db.col.createIndex({content: "text"})
地理空间索引 支持 2d（平面坐标）和 2dsphere（球面坐标）地理空间查询 db.col.createIndex({loc: "2dsphere"})
哈希索引 对字段值进行哈希后建立索引，常用于哈希分片的分片键 db.col.createIndex({shardKey: "hashed"})
通配符索引 对未知或动态字段名建立索引，使用 $** 匹配所有字段 db.col.createIndex({"$**": 1})
集群化索引 文档按索引键顺序物理存储，无需额外索引结构，查询时直接读取 createCollection with clusteredIndex
限制：单个集合最多支持 64 个索引。复合索引最多包含 32 个字段。一个集合只能有一个文本索引。

3. 复合索引键顺序 (ESR 规则)

复合索引中字段的排列顺序直接影响索引能否被查询有效利用。ESR 规则（Equality-Sort-Range）是 MongoDB 复合索引设计的核心原则。

ESR 含义

  • E -- Equality（等值匹配）：查询条件中使用精确匹配的字段，如 {status: "active"}
  • S -- Sort（排序）：查询中需要排序的字段，如 .sort({createdAt: -1})
  • R -- Range（范围查询）：查询条件中使用范围操作符的字段，如 {age: {$gte: 18}}

最优字段排列

E - 等值匹配
精确过滤，缩小区间
优先级 1 -- 最先排列
→
S - 排序
避免内存排序
优先级 2 -- 其次排列
→
R - 范围查询
有序扫描范围
优先级 3 -- 最后排列

示例

查询条件：{status: "active", age: {$gte: 18}}，排序：.sort({name: 1})

最优复合索引：{status: 1, name: 1, age: 1}

  • status 放第一 -- 等值匹配，快速定位
  • name 放第二 -- 排序字段，利用索引有序性
  • age 放第三 -- 范围查询，有序扫描

索引前缀

复合索引 {a: 1, b: 1, c: 1} 包含以下前缀索引：

  • {a: 1}
  • {a: 1, b: 1}
  • {a: 1, b: 1, c: 1}（完整索引本身）

查询条件只涉及前缀字段时，可以复用复合索引，无需额外创建单字段索引。这有助于减少索引总数，降低写入开销。

技巧：将等值匹配字段排在前面可以最大限度缩小索引扫描范围，让后续的排序和范围操作在更小的数据集上执行，从而提升整体查询效率。

4. 索引属性

索引属性是对索引行为的额外约束或扩展，可叠加在不同索引类型上，实现更精细的控制。

属性 说明 兼容类型
唯一索引 Unique 拒绝索引字段中存在重复值。在复合索引中使用时，字段组合值必须唯一（单个字段可重复） 单字段 / 复合索引
TTL 索引 TTL 自动删除超过指定时间的文档。需指定 expireAfterSeconds，仅适用于 Date 类型字段 单字段索引 (Date)
部分索引 Partial 仅索引满足指定条件的文档（通过 partialFilterExpression），减少索引大小和维护开销 所有索引类型
稀疏索引 Sparse 仅索引包含索引字段的文档，跳过字段缺失或为 null 的文档 所有索引类型
隐藏索引 Hidden 对查询规划器不可见，不会影响查询计划选择。可用于安全评估删除索引的影响 所有索引类型
不区分大小写 Collation 通过指定排序规则（Collation）使索引在比较和排序时忽略大小写 字符串字段索引

属性组合示例

// 唯一 + 部分索引：仅对活跃用户强制唯一
db.users.createIndex(
  { email: 1 },
  {
    unique: true,
    partialFilterExpression: { status: "active" }
  }
)

// TTL 索引：日志保留 30 天
db.logs.createIndex(
  { createdAt: 1 },
  { expireAfterSeconds: 2592000 }
)

// 隐藏索引：测试删除索引的影响
db.orders.createIndex(
  { customerId: 1 },
  { hidden: true }
)

5. 索引管理

创建索引

// 创建单个索引
db.collection.createIndex({ field: 1 }, options)

// 同时创建多个索引
db.collection.createIndexes([
  { key: { name: 1 } },
  { key: { age: -1, status: 1 } }
])

// 常用 options
{
  unique: true,                // 唯一约束
  name: "myIndexName",        // 自定义索引名
  background: true,            // (4.2前) 后台构建
  partialFilterExpression: {}, // 部分索引条件
  sparse: true,               // 稀疏索引
  expireAfterSeconds: 3600,   // TTL
  hidden: true,                // 隐藏索引
  collation: { locale: "zh", strength: 2 } // 排序规则
}

删除索引

// 删除指定索引（按名称）
db.collection.dropIndex("indexName")

// 删除指定索引（按键模式）
db.collection.dropIndex({ name: 1 })

// 删除所有非 _id 索引
db.collection.dropIndexes()

查看索引

// 查看集合上的所有索引
db.collection.getIndexes()

// 查看索引使用统计
db.collection.aggregate([
  { $indexStats: {} }
])

索引命名规则

默认索引名称由字段名和排序方向拼接而成：

  • 单字段：{name: 1} → name_1
  • 复合索引：{name: 1, age: -1} → name_1_age_-1
  • 自定义名称：name: "idx_user_name"
注意：索引名称一旦创建不可修改，只能删除重建。建议在创建时指定有意义的名称，尤其是复合索引。

6. 索引构建过程

构建方式

从 MongoDB 4.2 开始，索引构建统一使用优化的构建过程（之前版本分为前台构建和后台构建两种模式）。

阶段 1：初始化与快照

获取集合的当前数据快照，开始扫描文档。此阶段会短暂持有集合的排他锁（仅影响开始瞬间）。

阶段 2：索引键构建

扫描集合中的所有文档，提取索引键值并排序。此阶段允许并发读写操作，新写入的数据通过 oplog 追踪。

阶段 3：应用变更

将构建期间发生的写入操作（插入、更新、删除）应用到正在构建的索引上。此阶段可能影响写入性能。

阶段 4：提交索引

索引构建完成后提交，索引变为可用状态。查询规划器开始考虑使用该索引。

对性能的影响

  • CPU 和内存：构建索引需要额外的 CPU 资源进行排序，以及内存来缓存索引条目
  • 磁盘 I/O：大量读写操作，可能影响其他操作的 I/O 性能
  • 副本集：在主节点上构建索引时，secondary 会自动同步构建（5.0+ 支持可同时构建）
最佳实践：在大集合上构建索引时，建议在低峰期进行。可以使用 commitQuorum 参数控制副本集上的构建提交条件。在 MongoDB 4.4+ 中，可以在 secondary 上优先构建，再提升为主节点。

7. 索引策略

合理的索引策略需要在查询性能和写入开销之间找到平衡点。以下是关键策略建议：

遵循 ESR 规则

复合索引中，将等值匹配字段排在前面，排序字段居中，范围查询字段放最后。确保索引最左前缀能被最大化利用。

核心原则

覆盖查询

当查询所需的所有字段（包括返回字段和条件字段）都包含在索引中时，MongoDB 可以仅从索引返回结果，无需访问文档本身，极大提升性能。

性能优化

避免多余索引

每个索引都增加写入开销和存储成本。利用 $indexStats 识别未使用或低效的索引。重复前缀的复合索引可以合并。

精简设计

使用 explain() 验证

通过 db.col.find(...).explain("executionStats") 检查查询是否命中索引、扫描了多少文档、执行耗时等。关注 totalDocsExamined 与 nReturned 的比例。

调试工具

覆盖查询示例

// 索引: { status: 1, name: 1 }

// 覆盖查询 -- 仅从索引返回结果
db.users.find(
  { status: "active" },
  { name: 1, _id: 0 }   // 只返回索引包含的字段
)

// 非覆盖查询 -- 需要回表获取额外字段
db.users.find(
  { status: "active" },
  { name: 1, email: 1, _id: 0 }  // email 不在索引中
)

explain() 关键指标

指标 含义 理想值
winningPlan.stage 查询阶段类型 IXSCAN（而非 COLLSCAN）
totalDocsExamined 扫描文档数 接近 nReturned
executionTimeMillis 执行时间 尽可能低
indexBounds 索引扫描范围 有明确边界（非 MinKey~MaxKey）
isMultiKey 是否使用多键索引 根据预期判断

8. B-Tree 结构图解

以下展示 MongoDB 索引底层 B-Tree 结构的简化模型。根节点和中间节点存储键值与子节点指针，叶子节点存储实际索引键和文档引用（RecordId）。

B-Tree 索引结构（简化示例）

30
10 | 20
40 | 50
1|5|10
12|15|20
30|35|40
45|50|55
叶子节点：索引键值 + RecordId → 指向文档物理位置

B-Tree 查询过程

1. 从根节点开始，比较目标值与节点键值
↓
2. 选择对应的子节点指针，进入下一层
↓
3. 重复比较直到到达叶子节点
↓
4. 在叶子节点中找到索引键，获取 RecordId
↓
5. 通过 RecordId 定位并返回文档数据
关键优势：无论集合有多少文档，B-Tree 查询深度通常只有 3-4 层，保证了 O(log n) 的查询时间复杂度。相比全集合扫描 O(n)，在大数据集上性能差异可达数个数量级。
