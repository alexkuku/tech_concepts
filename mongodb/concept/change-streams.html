变更流

Change Streams - 实时数据变更监听

概述

变更流（Change Streams）允许应用程序实时访问集合、数据库或整个部署的数据变更，而无需手动跟踪 oplog（操作日志）。变更流基于 MongoDB 的复制机制，将数据变更事件以结构化的文档形式推送给监听者。

应用程序
| watch() 订阅变更
Change Stream API
| 读取 oplog
副本集 Primary
| 写入 oplog
Oplog (操作日志)

变更流将 oplog 中的原始操作转化为易于消费的事件文档格式，包含操作类型、完整文档、命名空间等结构化信息。

前提条件

使用变更流需要满足以下条件：

条件 说明
存储引擎 必须使用 WiredTiger 存储引擎
复制协议版本 必须使用 pv1（复制协议版本 1）
读关注级别 必须使用 "majority" 读关注（read concern）
部署模式 副本集或分片集群（不支持单机实例）
注意：单机实例（standalone）不支持变更流。必须在副本集或分片集群环境中运行。

监听目标

变更流支持三种粒度的监听目标，从细粒度到粗粒度：

集合级别
db.collection.watch()
数据库级别
db.watch()
部署级别
Mongo.watch()
监听范围递增 →
监听目标 方法 监听范围 可用版本
集合 db.collection.watch() 单个集合的数据变更 3.6+
数据库 db.watch() 该数据库中所有集合的变更 4.0+
部署 Mongo.watch() 整个部署（除 config/local/admin 外） 4.0+

管道阶段

变更流支持使用聚合管道阶段来修改和过滤输出。可以在 watch() 方法中传入管道数组。

支持的管道阶段

阶段 作用 示例
$addFields 向变更事件文档添加新字段 添加处理时间戳等派生字段
$match 过滤变更事件，仅传递匹配的文档 仅监听特定命名空间或操作类型
$project 重塑文档，包含/排除指定字段 仅保留需要的字段以减少数据传输
$replaceRoot 用指定文档替换根文档 将 fullDocument 提升为根文档
$replaceWith 同 $replaceRoot（别名） 同上
$redact 基于字段值限制文档内容 基于安全策略裁剪敏感字段
$set 设置文档中的字段值（同 $addFields 别名） 添加或覆盖字段
$unset 移除文档中的指定字段 排除不需要的元数据字段
提示：使用 $match 在管道中尽早过滤变更事件，可以减少不必要的事件处理开销，提高性能。

管道示例

// 仅监听 insert 操作
db.orders.watch([
  { $match: { operationType: "insert" } }
])

// 仅监听特定用户集合的变更并添加字段
db.users.watch([
  { $match: { "fullDocument.status": "active" } },
  { $addFields: { processedAt: "$$NOW" } }
])

完整文档查找

默认情况下，对于 update 操作，变更流仅返回增量变更（差异字段），不包含完整文档。通过设置 fullDocument 选项可以获取更新后的完整文档。

选项值 行为 适用操作
"updateLookup" 在更新操作时，额外查询集合以获取更新后的完整文档 update
"whenAvailable" 如果后置镜像可用则返回，否则返回 null update, replace
"required" 必须返回后置镜像，不可用则报错 update, replace 
// 使用 updateLookup 获取完整文档
const changeStream = db.collection("orders").watch(
  [],
  { fullDocument: "updateLookup" }
);

changeStream.on("change", (event) => {
  console.log(event.fullDocument); // 包含更新后的完整文档
});
注意："updateLookup" 是在变更发生之后发起的额外查询，因此返回的文档可能不反映变更时的精确状态，如果在查询与变更之间有其他更新发生。

恢复令牌

变更流支持断点续传功能，在连接中断后可以从上次的位置继续消费变更事件。每个变更事件都包含一个 _id 字段（即恢复令牌 / resume token）。

开始监听变更流
↓
接收变更事件，保存 resume token
↓
连接中断
↓
使用 resumeAfter / startAfter 恢复
↓
从断点继续消费变更

两种恢复方式对比

特性 resumeAfter startAfter
参数 resume token resume token
跳过指定事件 是（从指定 token 之后开始） 是（从指定 token 之后开始）
invalidate 事件后可恢复 不可 可以
适用场景 常规断点续传 invalidate 事件后恢复监听
可用版本 3.6+ 4.1.1+
说明：当监听的集合被删除（drop）或重命名时，会触发 invalidate 事件，此事件之后变更流关闭。使用 resumeAfter 无法恢复，必须使用 startAfter 从 invalidate 事件的 token 之后重新打开新的变更流。 
// resumeAfter 示例
let lastToken = null;

changeStream.on("change", (event) => {
  lastToken = event._id;  // 保存恢复令牌
  processEvent(event);
});

// 连接恢复后
const newStream = db.collection("orders").watch([], {
  resumeAfter: lastToken
});

// startAfter 示例 - 在 invalidate 事件后恢复
const newStream = db.collection("orders").watch([], {
  startAfter: invalidateToken
});

前置与后置镜像

从 MongoDB 6.0 开始，变更流支持前置镜像（Pre-Image）和后置镜像（Post-Image），可以在变更事件中获取文档变更前和变更后的完整状态。

变更事件文档
|
fullDocument (后置镜像)
变更后的文档状态
fullDocumentBeforeChange (前置镜像)
变更前的文档状态
|
集合设置
changeStreamPreAndPostImages

启用前置与后置镜像

// 在集合级别启用
db.createCollection("orders");
db.runCommand({
  collMod: "orders",
  changeStreamPreAndPostImages: { enabled: true }
});

// 监听时指定选项
db.orders.watch([], {
  fullDocumentBeforeChange: "whenAvailable",  // 或 "required"
  fullDocument: "whenAvailable"               // 或 "required"
});

fullDocumentBeforeChange 选项

选项值 行为
"whenAvailable" 前置镜像可用时返回，不可用时返回 null
"required" 前置镜像必须可用，不可用时产生错误
注意：前置与后置镜像会占用额外的存储空间，并且有过期时间（通过 expireAfterSeconds 配置）。请根据业务需求合理设置保留时间。

代码示例

Python 示例

import pymongo

client = pymongo.MongoClient("mongodb://localhost:27017/?replicaSet=rs0")
db = client["mydb"]
collection = db["orders"]

# 基本变更流
with collection.watch() as stream:
    for change in stream:
        print(f"操作类型: {change['operationType']}")
        print(f"命名空间: {change['ns']}")
        if "fullDocument" in change:
            print(f"完整文档: {change['fullDocument']}")
        print(f"恢复令牌: {change['_id']}")

# 带管道过滤的变更流
pipeline = [
    {"$match": {"operationType": {"$in": ["insert", "update"]}}},
    {"$addFields": {"processedAt": "$$NOW"}}
]

with collection.watch(pipeline, full_document="updateLookup") as stream:
    for change in stream:
        print(change)

# 断点续传
resume_token = None

with collection.watch() as stream:
    for change in stream:
        resume_token = stream.resume_token
        process_change(change)

# 恢复变更流
with collection.watch(resume_after=resume_token) as stream:
    for change in stream:
        process_change(change)

Node.js 示例

const { MongoClient } = require("mongodb");

async function watchChanges() {
  const client = new MongoClient("mongodb://localhost:27017/?replicaSet=rs0");
  await client.connect();
  const db = client.db("mydb");
  const collection = db.collection("orders");

  // 基本变更流
  const changeStream = collection.watch();

  changeStream.on("change", (event) => {
    console.log(`操作类型: ${event.operationType}`);
    console.log(`命名空间: ${event.ns}`);
    if (event.fullDocument) {
      console.log(`完整文档: ${JSON.stringify(event.fullDocument)}`);
    }
  });

  // 带管道过滤
  const pipeline = [
    { $match: { operationType: { $in: ["insert", "update"] } } },
    { $project: { "fullDocument.status": 1, operationType: 1 } }
  ];

  const filteredStream = collection.watch(pipeline, {
    fullDocument: "updateLookup"
  });

  filteredStream.on("change", (event) => {
    console.log("过滤后的事件:", event);
  });

  // 错误处理与恢复
  changeStream.on("error", async (error) => {
    console.error("变更流错误:", error);
    // 等待后重试
    await new Promise(resolve => setTimeout(resolve, 5000));
    // 使用 resumeAfter 恢复
    const newStream = collection.watch([], {
      resumeAfter: lastResumeToken
    });
  });

  // 优雅关闭
  process.on("SIGINT", async () => {
    await changeStream.close();
    await client.close();
    process.exit(0);
  });
}

watchChanges().catch(console.error);

应用场景

场景 说明 典型实现
ETL 数据同步 将 MongoDB 变更实时同步到数据仓库、搜索引擎或其他数据存储 监听变更流 → 转换数据 → 写入目标系统
跨平台数据同步 在 MongoDB 与 Redis、Elasticsearch 等之间保持数据一致 变更流 + 消息队列（Kafka/RabbitMQ）

性能考量

考量项 建议 原因
连接池大小 maxPoolSize 应大于变更流数量 每个变更流占用一个连接，连接池不足会导致等待
$lookup 使用 谨慎使用 $lookup 每次变更事件都触发关联查询，开销较大
管道过滤 尽早使用 $match 过滤 减少下游处理的事件数量
事件处理速度 确保事件消费速度 >= 生产速度 处理过慢会导致 oplog 积压，可能被覆盖
fullDocument 开销 仅在必要时启用 updateLookup 额外查询会增加服务器负载
分片集群 通过 mongos 监听变更流 mongos 会合并所有分片的变更事件
最佳实践：在生产环境中，始终为变更流实现错误处理和自动恢复机制，确保在连接断开后能够通过 resume token 恢复监听。
