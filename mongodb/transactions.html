MongoDB 事务

多文档 ACID 事务与分布式数据一致性

事务概述

在 MongoDB 中，单文档操作天然具有原子性 —— 对单个文档的任何写操作（包括对嵌入文档和数组的更新）都是原子的，要么全部成功，要么全部回滚，无需显式开启事务。

然而，当操作涉及多个文档或多个集合时，需要使用多文档 ACID 事务来保证数据的完整性和一致性。MongoDB 从 4.0 版本开始支持副本集上的多文档事务，从 4.2 版本开始支持分片集群上的分布式事务。

核心要点：
  • 单文档操作是原子性的，无需事务
  • 多文档事务适用于跨文档、跨集合的复杂操作
  • 分布式事务可在副本集和分片集群上运行
  • 事务提供完整的 ACID 保证

ACID 保证

MongoDB 多文档事务提供完整的 ACID 保证，确保数据在并发和故障场景下的一致性：

A
原子性 Atomicity
事务中的所有操作要么全部成功提交，要么全部失败回滚，不存在部分完成的状态。
C
一致性 Consistency
事务执行前后，数据库始终处于一致状态，所有约束和规则（唯一索引、文档验证等）均得到维护。
I
隔离性 Isolation
并发执行的事务相互隔离，不会互相干扰。MongoDB 使用快照隔离确保事务看到一致的数据视图。
D
持久性 Durability
一旦事务提交，其修改将持久保存，即使发生系统故障也不会丢失。配合 w:majority 写关注确保持久性。

事务 API

MongoDB 提供两种事务 API，开发者应根据场景选择合适的方式：

Callback API（推荐）

  • 自动重试 TransientTransactionError
  • 自动重试 UnknownTransactionCommitResult
  • 包含提交重试逻辑
  • 代码更简洁，减少手动错误处理
  • 适合大多数应用场景

Core API（核心 API）

  • 需要手动重试事务
  • 需要手动重试提交
  • 提供更细粒度的控制
  • 错误处理需自行实现
  • 适合需要精确控制的场景
关键规则：事务中的每个操作都必须传入 session 参数，否则操作将不在事务上下文中执行。

Python 示例：Callback API

Python
from pymongo import MongoClient
from pymongo.write_concern import WriteConcern
from pymongo.read_concern import ReadConcern

client = MongoClient("mongodb://localhost:27017")
wc_majority = WriteConcern("majority")
rc_snapshot = ReadConcern("snapshot")

# Callback 函数：包含事务中要执行的操作
def transfer_funds(session, from_acc, to_acc, amount):
    db = client.get_database(
        "bank",
        read_concern=rc_snapshot,
        write_concern=wc_majority
    )

    # 扣款 - 注意传入 session
    db.accounts.update_one(
        {"_id": from_acc},
        {"$inc": {"balance": -amount}},
        session=session
    )

    # 入账 - 注意传入 session
    db.accounts.update_one(
        {"_id": to_acc},
        {"$inc": {"balance": amount}},
        session=session
    )

# 使用 with_transaction 自动管理事务
with client.start_session() as session:
    session.with_transaction(
        transfer_funds,
        args=(session, "A001", "A002", 500),
        read_concern=rc_snapshot,
        write_concern=wc_majority
    )

print("转账完成")

Python 示例：Core API

Python
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017")

# 手动管理事务生命周期
with client.start_session() as session:
    session.start_transaction(
        read_concern={"level": "snapshot"},
        write_concern={"w": "majority"}
    )

    try:
        db = client.bank

        # 操作必须传入 session
        db.accounts.update_one(
            {"_id": "A001"},
            {"$inc": {"balance": -500}},
            session=session
        )

        db.accounts.update_one(
            {"_id": "A002"},
            {"$inc": {"balance": 500}},
            session=session
        )

        # 手动提交
        session.commit_transaction()
        print("事务提交成功")

    except Exception as e:
        # 手动中止
        session.abort_transaction()
        print(f"事务已回滚: {e}")

事务操作一览

并非所有操作都允许在事务中执行，下表列出了事务中允许和禁止的操作：

类别 操作 说明
允许 CRUD 操作 insert / update / delete / find 均可在事务中执行
允许 创建集合与索引 仅在非跨分片写事务中支持
允许 Count / Distinct 通过 $count / $group 聚合阶段实现
禁止 跨分片创建集合 事务中不能创建新的分片集合
禁止 写入固定集合 固定集合（Capped Collection）为只追加模式，不支持事务写操作
禁止 分片上的 $graphLookup 在分片集群事务中不支持 $graphLookup 阶段

会话与事务

事务与会话（Session）紧密关联，理解它们的关系对正确使用事务至关重要：

  • 一个会话同一时间只能有一个事务 —— 不能在同一个会话中嵌套事务或并行开启多个事务
  • 会话结束时事务自动中止 —— 如果会话关闭时还有未提交的事务，该事务将被自动回滚
  • 不同会话可以各自独立运行事务，实现并发
  • 建议使用 with 语句管理会话生命周期，确保资源正确释放
注意：不要在事务进行中丢弃会话对象。如果会话被垃圾回收而事务尚未提交，事务将自动中止，所有修改将回滚。

读写关注与读偏好

读偏好（Read Preference）

事务中的所有操作必须路由到同一个成员，因此读偏好必须为 primary。这确保事务中读写操作在同一节点上执行，避免跨节点数据不一致。

读关注级别（Read Concern Levels）

local 读取最近已写入的数据，不保证已提交到大多数节点。性能最高，一致性最弱。
majority 只读取已被大多数节点确认的数据。提供较强的读一致性保证。
snapshot 读取事务开始时刻的一致性快照。这是事务中最常用的读关注级别，提供完整的快照隔离。

写关注（Write Concern）

w:1 仅主节点确认写入。提交后如果主节点故障，数据可能回滚。不保证持久性。
w:majority 大多数节点确认写入。提交后即使主节点故障，数据也不会丢失。事务中推荐使用。

优先级继承

读写关注和读偏好可以在多个层级设置，优先级从高到低为：

事务生命周期

一个完整的事务从开始到结束经历以下阶段：

开始事务
startTransaction
创建事务上下文
执行操作
CRUD
传入 session 参数
提交事务
commitTransaction
持久化所有修改
或
执行操作
CRUD
中止事务
abortTransaction
回滚所有修改
生命周期要点：事务提交后，所有修改对所有客户端可见。事务中止后，所有修改被丢弃，其他客户端不受影响。如果事务既未提交也未中止，会话结束时将自动中止。

单文档原子性 vs 多文档事务

MongoDB 的单文档原子性和多文档事务各有适用场景，理解差异有助于做出正确的架构决策：

特性 单文档原子性 多文档事务
原子性 内建保证，无需显式操作 需显式开启事务
性能影响 无额外开销 有额外开销（锁、日志、协调）
隔离级别 文档级隔离 快照隔离
适用场景 嵌入式文档模型，单文档内更新 跨集合、跨文档的复杂操作
锁粒度 文档级锁 多文档锁，可能影响并发
重试复杂度 简单，单操作重试 复杂，需整体重试事务
推荐程度 优先使用 按需使用
设计建议：优先使用嵌入式文档模型，将相关数据放在同一个文档中，利用单文档原子性避免事务开销。只有在操作确实需要跨文档/跨集合时，才使用多文档事务。

重要注意事项

使用事务前务必了解
  • 事务具有更高的性能成本，包括网络往返、锁管理和操作日志记录，应仅在必要时使用
  • 事务不能替代良好的 Schema 设计，合理的数据建模仍是首选方案
  • 反范式化（嵌入式文档/数组）仍是许多场景的最佳选择，可避免跨文档事务
  • 副本集事务要求 FCV（Feature Compatibility Version）>= 4.0
  • 分片集群事务要求 FCV >= 4.2
  • 主节点必须使用 WiredTiger 存储引擎
  • 事务默认超时时间为 60 秒，可通过 transactionLifetimeLimitSeconds 参数调整
  • 事务中创建索引是 abort 操作，会导致事务回滚
性能提示：长时间运行的事务会持有锁并消耗更多资源，应尽量缩短事务的执行时间。如果操作可以在事务外完成（如读操作），应放在事务之外执行。
版本要求：
  • 副本集多文档事务：MongoDB >= 4.0
  • 分片集群分布式事务：MongoDB >= 4.2
  • 如果升级后 FCV 未更新，事务功能不可用
